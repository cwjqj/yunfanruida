## 1. 变更目的

顶装人数统计不再默认使用脚本 confirmed-track 数量，而是等待雷达 `0x86/0x0C` 精准人数：

- 精准人数 `min=max` 才更新最终 `people_count`。
- 实时人数 `0x86/0x0A` 只作预览并触发 pending。
- 门事件 `0x86/0x14` 只记录方向并触发 pending，不直接加减人数。
- confirmed tracks 保留为 `track_people_count`、稳定轨迹和回退模式。

修改前脚本已备份至：

```text
顶装/备份/20260714_145119_before_accurate_people_count/radar_iot_task.lua
```

## 2. 日志证据

用户提供的约 14:22:39 至 14:32:48 日志摘要显示：

- 单人阶段原始目标偶尔短暂增加，但精准人数保持 1。
- 多人阶段轨迹帧长度 `0x4D/0x58` 对应 7/8 个 11B 目标，实时/精准人数约为 6~8。
- 进出门事件与人数变化同时出现。

同时，起始时存在“精准人数 1、实时人数和轨迹为 2”的情况，说明精准人数不是即时预览值，而是需要等待的新确认结果。因此代码保留最后 accepted 精准人数，并通过 `people_count_pending` 表示当前是否存在更新等待。

本工程目录中未找到该时间段的原始日志文件，本节依据用户提供的摘要记录；原始时间戳、完整 HEX 和人工真值仍需后续保存并复核。

## 3. 新增状态字段

| 字段 | 规则 |
|---|---|
| `people_count_valid` | 收到未过期且 `min=max` 的精准人数后为 1；90 秒未更新后为 0 |
| `people_count_pending` | 首次等待、实时冲突、门事件、精准区间或精准过期时为 1 |
| `people_count_source` | 默认 `accurate` |
| `track_people_count` | confirmed tracks 数量，仅作对照 |
| `accurate_age_sec` | 最后精准帧年龄；未收到或雷达超时为 -1 |
| `door_event` | -1=无，0=进，1=出 |
| `door_event_count` | 合法门事件累计数 |
| `door_event_at` | 最近门事件的秒时间戳 |

## 4. 离线测试

测试文件：`顶装/脚本/lua/tests/count/test_radar_iot_task.lua`

运行：

```powershell
cd .\顶装\脚本\lua
lua .\tests/count/test_radar_iot_task.lua
```

测试序列：

1. 实时人数 2，尚无精准帧。
2. 精准人数 1，但实时人数仍为 2，最终人数为 1 且保持 pending。
3. 收到出门事件，人数不直接递减。
4. 实时人数回到 1。
5. 下一条精准人数 1 清除 pending。
6. 精准区间 `[1,2]` 不覆盖上一稳定人数。
7. 实时人数 2、精准人数 2 后接受最终人数 2。
8. 持续心跳但 95 秒无精准人数：保留 2，设置 valid=0、pending=1。
9. 独立场景中雷达整体断帧超过 30 秒：人数清零、`radar_ready=0`。

测试输出：

```text
PASS: 103 frames, 103 dup reports, final accurate age=95
PASS: radar stale scenario cleared count (42 simulated seconds total)
```

附加检查：

- DTU 裸 `function` 包装在内存中替换为命名函数后，标准 Lua `load()` 语法检查通过。
- 测试脚本通过 `luac -p`。
- 最终人数不存在由 `target_count` 或实时人数直接赋值的路径。
