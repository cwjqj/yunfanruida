## 1. 工程边界与统一主线

### 1.1 工程做什么

```text
R60BMP1 雷达
    │ UART TTL 3.3V，115200 8N1
    ├── PC 端 Python：查询、设置、抓包、协议解析
    │
    └── Air780EP DTU：读取 UART1 → 解析 → 计算 → MQTT 上云
                                      │
                                      └── 银尔达/YED IoT 物模型数据点
```

统一处理链路如下：

1. 确认硬件接线和供电。
2. 用 PC 串口工具查询雷达内部参数，保存读回结果。
3. 根据安装场景设置角度、高度、探测范围和功能开关。
4. 再次查询并逐项校验，确认“写入值”和“读回值”一致。
5. 将当前 DTU Lua 主脚本部署到云平台任务中。
6. 观察雷达原始帧、Lua 解析结果和云平台数据点，按测试记录完成现场验证。

## 3. 雷达通信协议与解析规则

### 3.1 通用帧格式

```text
53 59 | Control | Command | Len_H Len_L | Data... | Sum | 54 43
```

| 字段 | 长度 | 规则 |
|---|---:|---|
| 帧头 | 2B | 固定 `53 59` |
| 控制字 | 1B | 数据类别 |
| 命令字 | 1B | 读、写或主动上报的具体命令 |
| 数据长度 | 2B | 大端序，无符号 |
| 数据区 | nB | 由控制字/命令字决定 |
| 校验和 | 1B | 从帧头开始到数据区结束，逐字节求和后取低 8 位 |
| 帧尾 | 2B | 固定 `54 43` |

解析器必须先找帧头，再按长度等待完整帧，最后校验帧尾和校验和；串口一次读取可能包含半帧、多个帧或前导噪声，不能假设“一次 read 对应一帧”。

### 3.2 当前工程使用的控制字

| 控制字 | 含义 | 当前解析/查询内容 |
|---|---|---|
| `0x01` | 心跳 | 只做链路参考，不作为主要业务数据 |
| `0x02` | 产品信息 | 固件版本 |
| `0x05` | 工作状态 | 初始化完成状态 |
| `0x06` | 安装信息 | 安装角度、安装高度 |
| `0x07` | 人数统计配置 | 探测范围、自动范围、标签、干扰边 |
| `0x80` | 人体存在 | 有人/无人、运动、体动 |
| `0x82` | 轨迹跟踪 | 目标坐标、高度、速度 |
| `0x86` | 人数统计 | 实时人数、精准人数和相关时间参数 |

### 3.3 轨迹目标解析

当前项目按每个目标 11 字节解析：

```text
id(1) + size(1) + feature(1) + x(2) + y(2) + height(2) + speed(2)
```

坐标和速度采用雷达的“符号位”格式：最高位表示负号，其余位表示绝对值；不能直接当作普通 int16 补码。Lua 和 Python 解析器必须保持同一规则。

已验证主动轨迹帧示例：

```text
53 59 82 02 00 0B 00 01 02 80 08 00 2E 00 B8 00 06 B2 54 43
```

解析结果：目标 ID `0`，X=`-8cm`，Y=`46cm`，高度=`184cm`，速度=`6cm/s`。

### 4.4 设置顺序

```text
安装角度 → 安装高度 → 自动范围/固定边界 → 干扰边 → 人体存在 → 轨迹跟踪 → 人数统计参数 → 查询回读
```

每一步都应遵循“发送设置帧 → 等待响应 → 查询回读 → 比较期望值”。侧装、顶装的一键脚本只负责场景参数组合，底层协议仍以通用工具和本文为准。

### 4.5 参数冲突处理

- PC 查询与 DTU Lua 同时占用 UART 时，可能互相消费回复；查询时应临时停止 DTU 任务。
- `07/88` 自动范围限制在部分记录中无响应，不能仅凭“发送成功”判断生效。
- 安装角度、高度、边界和标签属于雷达内部配置，不由当前 DTU 主脚本自动写入；修改后必须重新查询。

## 5. 云端 Lua 文件框架与当前内容

### 5.1 文件归属

当前可直接部署的 DTU 主脚本位于：

```text
侧装/脚本/lua/radar_iot_task.lua
```

安装方式云端切换任务位于：

```text
通用/脚本/lua/radar_install_mode_task.lua
```

可嵌入现有主任务的无主循环模块位于：

```text
通用/脚本/lua/radar_install_mode_module.lua
```

详细接入、状态清理、云端回执和实物验证步骤见：

```text
通用/文档/安装模式云端切换模块说明.md
```

推荐后续任务复用 `radar_install_mode_module.lua`。该模块不读取 UART、不读取网络、不创建主循环，只提供 `request(mode, did)`、`handle_frame(frame)` 和 `update(now)` 三个接入口；主任务继续独占自己的串口/网络读取，并通过回调清除业务残留、切换场景参数和恢复上报。若 DTU 环境不支持 `require`，可将该模块内容复制到主任务的辅助函数区域，保留相同 API。

该任务通过云平台开关量数据点 `mode` 设置雷达模块内部安装参数，并在查询读回一致后才返回成功。配置开始时先清除旧状态与 UART 残留，再严格按“安装角度 → 安装高度 → 清空配置文件探测范围 → 场景边界 → 自动范围模式”逐项写入和回读。它是独立示例任务，会独占 UART1 和网络通道 1 的接收队列；现有顶装和侧装主任务不自动启用它，后续新任务应优先复用无主循环模块。

目录不移动。之所以放在侧装目录，是因为当前主脚本包含寝室上床下桌的区域和人数规则；其中的**串口帧解析、状态上报、云平台数据点结构**可作为通用框架复用。

同目录其他 Lua 文件：

| 文件 | 定位 |
|---|---|
| `radar_iot_task.lua` | 当前主脚本，部署入口 |
| `radar_people_count_core.lua` | 核心人数统计版，便于对照解析和统计逻辑 |
| `radar_iot_task_track.lua` | 轨迹保持实验版 |
| `radar_iot_task_original.lua` | 最早原始版本，只作历史备份 |

### 5.2 主脚本的固定骨架

```text
1. 任务元信息与场景常量
2. UART1 / 网络通道初始化
3. 雷达帧缓冲与完整帧提取
4. 校验和、无符号/符号数、HEX 日志等基础函数
5. 0x80 人体存在/运动/体动解析
6. 0x82 轨迹目标解析与航迹关联
7. 0x86 实时/精准人数解析
8. 有效目标过滤、主目标选择、姿态判断
9. confirmed track 人数统计、保持、减人和兜底清零
10. 云平台数据点组装、变化检测和 MQTT 上报
11. 100ms 主循环、超时维护和异常日志
```

### 5.3 当前脚本内容的关键约定

- 启动时停止平台自动协议解析：`UartStopProRecCh(1)`、`PronetStopProRecCh(nid)`。
- 雷达原始帧先进入缓冲区，再按帧头、长度、帧尾、校验和提取。
- `0x80` 提供人体存在、运动和体动；`0x82` 提供目标轨迹；`0x86` 提供实时人数、精准人数和进出门事件。
- 人数口径按场景区分：当前侧装核心版以 confirmed tracks 为准；顶装 `Radar_Top_Count_1.2` 也使用软件范围内 confirmed tracks。顶装 1.1 的精准人数主源因边缘人员干扰而撤回，实时/精准人数和门事件均只作诊断。
- 航迹状态至少包含 active、holding 等状态，允许短时静止或丢帧后保持，再按出口、丢失、过期和看门狗规则减人。
- 每次上报同时保留 `radar_ready`、`radar_frame_count`、`uart_bytes` 等诊断字段，便于区分“无人”和“未收到有效帧”。

### 5.4 场景参数与通用代码的边界

以下参数属于侧装寝室场景，不应直接复制到所有安装方式：`AREA_X/Y`、出口区域、床下区域、目标高度过滤、航迹匹配距离、姿态阈值。顶装场景应以 `顶装/文档/顶装参数设置与验证.md` 和现场采样结果重新标定。

## 6. 云平台数据点统一定义

云平台数据点属于工程通用接口，放在本文统一维护；场景文档只记录现场如何验证这些字段。下面先按**当前部署版 Lua 是否实际输出**区分，避免云平台残留字段被误认为当前功能。

### 6.1 当前主脚本实际支持的数据点

以下字段都能在 `侧装/脚本/lua/radar_iot_task.lua` 的 `fill_param()` 中找到。

独立安装方式任务 `通用/脚本/lua/radar_install_mode_task.lua` 另支持以下可读写数据点：

| 标识 ID | 类型 | 含义 |
|---|---|---|
| `mode` | 开关量 | `0`=顶装（90°、330cm、X±165cm/Y±250cm、自动范围限制），`1`=侧装（30°、220cm、X±165cm/Y+=500cm/Y-=0cm、自动范围限制） |

`mode` 下发后，任务立即暂停业务数据更新并清空人数、航迹、目标、串口缓存等旧模式残留；随后执行下表流程，全部一致后才恢复业务解析、返回 `ssetbck rst=0` 并主动上报新值。非法值返回 `rst=1`，UART 发送或三次读回验证失败返回 `rst=2`，已有切换正在执行时返回 `rst=3`。

| 顺序 | 通用步骤 | 侧装 `mode=1` | 顶装 `mode=0` |
|---:|---|---|---|
| 1 | 安装角度 `0x06/0x01`，查询 `0x81` | Z=30° | Z=90° |
| 2 | 安装高度 `0x06/0x02`，查询 `0x82` | 220cm | 330cm |
| 3 | 配置文件范围 `0x07/0x17`，查询 `0x97` | 写入9字节全零，清除门/宽高/坐标点 | 同左 |
| 4 | 自动范围使用 `0x07/0x0C` | 在边界写入后开启 | 在边界写入后开启 |
| 5 | 探测边界 `0x07/0x09`，查询 `0x89` | X±165cm，Y+=500cm，Y-=0cm | X±165cm，Y±250cm |
| 6 | 自动范围限制 `0x07/0x08` | 最后开启 | 最后开启 |

配置文件范围的“删除”采用协议允许的数据结构写入9字节全零，并要求 `0x97` 回读仍为9字节全零才继续；这表示门中心、门宽高、范围类型和坐标点全部清空。

### 6.1.1 主任务复用接口

主任务只需要提供 UART 发送和状态清理回调：

```lua
local InstallMode=require("radar_install_mode_module")
local mode_controller=InstallMode.new({
    send_uart=function(data)
        -- Air780EP 优先直接写物理 UART；平台接口只作为回退。
        if type(uart)=="table" and type(uart.write)=="function" then
            return uart.write(1,data)
        end
        return UartSetSendCh(1,data)
    end,
    on_config_start=function(mode,profile)
        rx_buf=""
        clear_all("install_mode_start")
        radar_ready=0
    end,
    on_config_end=function(ok,mode,rst,reason)
        if ok then apply_scene_profile(mode) end
        needdup=1
    end
})
```

在主任务中按以下位置接入：

```lua
-- 完整帧校验通过后，优先交给安装参数模块
if mode_controller:handle_frame(frame) then return true end

-- 主循环每次调用
mode_controller:update(os.time())

-- 收到云端 sset 时
local mode=InstallMode.extract_mode(obj.param)
if mode==nil then
    -- 返回 ssetbck rst=1
else
    mode_controller:request(mode,obj.did)
end
```

`on_config_start` 是清除旧数据的扩展口；应清空 `tracks`、`target_count`、`target_ids`、主目标/轨迹字段、人数统计、`rx_buf`，并将 `radar_ready` 置为 `0`。`on_config_end` 成功后再加载顶装/侧装区域和阈值配置。现有顶装、侧装主任务不自动启用该模块，后续新任务可按上述接口自主复用。

#### 设备与版本

| 标识 ID | 类型 | 含义 |
|---|---|---|
| `imei` | 字符串 | DTU IMEI |
| `iccid` | 字符串 | SIM 卡 ICCID |
| `imsi` | 字符串 | SIM 卡 IMSI |

### 6.2 云平台已配置/历史文档存在，但当前主脚本未输出的数据点

下列字段在既有交接文档或云平台配置中出现过，但当前 `radar_iot_task.lua` 的 `fill_param()` 没有上报它们：

| 数据点 | 当前状态 | 风险/处理建议 |
|---|---|---|
| `posture_code` | 未输出 | 云端可能保留旧值；使用前需重新实现姿态计算和清零逻辑 |
| `primary_target_posture` | 未输出 | 同上 |
| `main_posture_text` | 未输出 | 同上 |
| `primary_in_bed_zone` | 未输出 | 依赖场景床区定义，不能作为纯通用字段直接计算 |
| `target0` / `target1` / `target2` | 未输出 | 当前核心版只上传主目标和航迹列表，没有组装复杂目标 JSON |

在当前部署版不增加代码的前提下，建议在云平台将这些字段标记为“预留/暂未启用”，或隐藏其展示；不要用旧值判断现场状态。

### 6.3 复杂目标数据点的预留格式

如果后续恢复 `target0`、`target1`、`target2`，每个数据点应使用 JSON 对象而不是数组：

```json
{
  "id": 1,
  "x": -20,
  "y": 180,
  "height": 168,
  "speed": 4,
  "posture": 2,
  "posture_text": "坐下",
  "in_bed_zone": 1
}
```

字段 `id/x/y/height/speed/posture/posture_text/in_bed_zone` 的单位和枚举必须与基础数据点保持一致。无目标时使用 `id=-1`，数值字段清零，避免云端继续显示上一目标。

### 6.4 上报原则

- 首次启动应主动上报一次完整状态。
- 常规循环按字段变化检测上报，避免每 100ms 重复发送。
- 顶装 1.3 的原始轨迹变化必须立即上报；完全相同的轨迹在第 1 帧和第 5 帧上报，随后仅每 30 秒保活。`radar_frame_count` 继续累计，但不单独触发上报。
- `radar_ready=0`、串口超时或看门狗清零时，应同步清理目标和人数，不能只停止上报。
- 业务 `people_count` 的最终口径必须在场景文档中明确：当前侧装核心版和顶装 1.3 均以范围内 confirmed tracks 为准；雷达实时/精准人数不能替代带软件 ROI 的人数。
- 云平台新增数据点后，必须先在 Lua `fill_param()` 中实现，再标记为“已启用”。

## 8. 推荐操作顺序与故障定位

### 8.1 新设备接入

1. 检查供电、电平、TX/RX 交叉和 GND。
2. 运行 `listen_radar.py`，确认能看到 `53 59` 开头的原始数据。
3. 运行 `query_radar_settings.py`，保存完整读回结果。
4. 使用通用设置工具或场景一键脚本下发参数。
5. 再次查询，逐项核对回读值。
6. 部署当前场景 Lua，先看 `radar_ready`、`uart_bytes` 和帧计数。
7. 再验证人数、轨迹、姿态和云端 JSON。

### 8.2 快速定位

| 现象 | 优先检查 |
|---|---|
| 完全无数据 | 供电、电平、TX/RX、COM 号、115200 8N1 |
| 有原始字节但无有效帧 | 帧头/长度/帧尾/校验和、是否半帧拼接 |
| PC 能读、DTU 无数据 | UART1 通道、平台自动协议解析是否已停止、Lua 是否运行 |
| `radar_ready=1` 但人数不对 | 轨迹 11B 解析、符号位规则、航迹关联和场景阈值 |
| 云端字段不更新 | 数据点 ID、类型、变化检测、MQTT 通道 `nid=1` |
| 查询无响应 | 是否被 DTU 占用串口、查询帧命令/校验是否正确 |
