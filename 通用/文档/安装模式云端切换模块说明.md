# 安装模式云端切换模块说明

更新时间：2026-07-15  
适用模块：云帆瑞达 R60BMP1 60G 毫米波雷达  
适用平台：Air780EP DTU、银尔达 / YED IoT 云平台

## 1. 文档用途

本文说明如何把云端安装模式切换功能加入现有 Lua 主程序。对应文件：

```text
通用/脚本/lua/radar_install_mode_module.lua
```

完整独立示例：

```text
通用/脚本/lua/radar_install_mode_task.lua
```

模块本身不读取 UART、不读取网络、不创建循环，因此不会与主程序争抢 `UartGetRecChAndDel()` 或 `PronetGetRecChAndDel()`。主程序继续负责通信，只在指定位置调用模块。

## 2. 云端数据点

云平台创建开关量数据点：

| 属性 | 设置 |
|---|---|
| 标识 ID | `mode` |
| 类型 | 开关量/整数 |
| 顶装 | `0` |
| 侧装 | `1` |
| 读写属性 | 可读、可写 |

常见下发消息：

```json
{
  "cmd": "sset",
  "did": "100",
  "param": {
    "mode": 1
  }
}
```

模块的 `extract_mode()` 同时兼容以下常见结构：

```lua
{mode=1}
{datapoint={mode=1}}
{id="mode",value=1}
{datapoint={{id="mode",value=1}}}
```

## 3. 两种模式的完整参数

| 参数 | 顶装 `mode=0` | 侧装 `mode=1` |
|---|---:|---:|
| X轴安装角度 | 0° | 0° |
| Y轴安装角度 | 0° | 0° |
| Z轴下倾角 | 90° | 30° |
| 安装高度 | 330cm | 220cm |
| X正边界 | 165cm | 165cm |
| X负边界 | 165cm | 165cm |
| Y正边界 | 250cm | 500cm |
| Y负边界 | 250cm | 0cm |
| 自动范围使用 | 开启 | 关闭 |
| 自动范围限制 | 开启 | 关闭 |

每次切换都先把配置文件探测范围 `0x07/0x17` 写成9字节全零，并通过 `0x07/0x97` 回读确认。这表示门中心、门宽、门高、范围类型和坐标点全部清空。

## 4. 配置状态机

### 4.1 通用步骤

```text
安装角度设置
    ↓ 0x06/0x81 回读
安装高度设置
    ↓ 0x06/0x82 回读
配置文件探测范围清零
    ↓ 0x07/0x97 回读
场景边界及自动范围配置
    ↓ 0x07/0x89 回读
全部成功后更新 mode 并返回成功
```

### 4.2 侧装流程

```text
Z=30°
→ 高度220cm
→ 清空配置文件探测范围
→ 关闭自动范围限制 0x07/0x08
→ 关闭自动范围使用 0x07/0x0C
→ 写固定边界 X±165、Y+=500、Y-=0
→ 回读验证
```

### 4.3 顶装流程

```text
Z=90°
→ 高度330cm
→ 清空配置文件探测范围
→ 写外边界 X±165、Y±250
→ 开启自动范围使用 0x07/0x0C
→ 最后开启自动范围限制 0x07/0x08
→ 回读验证
```

## 5. 模块接口

### 5.1 创建控制器

```lua
local InstallMode=require("radar_install_mode_module")
local clear_install_mode_residuals
local on_install_mode_finished

local mode_controller=InstallMode.new({
    send_uart=function(data)
        if type(uart)=="table" and type(uart.write)=="function" then
            return uart.write(1,data)
        end
        return UartSetSendCh(1,data)
    end,

    on_config_start=function(mode,profile)
        -- 配置开始前清除旧模式残留，见第6节。
        clear_install_mode_residuals()
    end,

    on_config_end=function(ok,mode,rst,reason)
        -- 这里只处理配置结果；云端回执由主程序发送。
        on_install_mode_finished(ok,mode,rst,reason)
    end,

    now=function()
        return os.time()
    end,

    max_attempts=3,
    verify_delay_sec=1,
    verify_timeout_sec=5
})
```

如果 DTU 环境不支持 `require()`，可把 `radar_install_mode_module.lua` 放入主任务辅助代码区域，并保持 `InstallMode.new()`、`request()`、`handle_frame()`、`update()` 的接口不变。

### 5.2 核心方法

| 方法 | 调用位置 | 作用 |
|---|---|---|
| `InstallMode.extract_mode(param)` | 收到云端 `sset` 时 | 从不同JSON结构中提取0或1 |
| `mode_controller:request(mode, token)` | 收到有效 `mode` 时 | 启动完整配置状态机 |
| `mode_controller:handle_frame(frame)` | 雷达完整帧校验通过后 | 消费安装参数和范围查询回复 |
| `mode_controller:update(os.time())` | 主循环每次迭代 | 驱动延时查询、超时和重试 |
| `mode_controller:is_busy()` | 接收新命令前 | 判断是否已有配置正在执行 |
| `mode_controller:get_mode()` | `sget`/主动上报时 | 获取最后一次完整成功的模式 |
| `mode_controller:get_pending_phase()` | 日志诊断 | 获取当前配置阶段 |

## 6. 配置开始时必须清除的数据

`on_config_start` 是主程序清除旧模式状态的扩展口。至少应清理：

```lua
clear_install_mode_residuals=function()
    -- 串口残留
    rx_buf=""

    -- 内部航迹
    tracks={}
    uid_seq=0
    all_clear_since=0

    -- 人数
    human_present=0
    has_person=0
    people_count=0
    people_min=0
    people_max=0
    target_count=0

    -- 主目标和兼容轨迹字段
    target_ids=""
    primary_target_id=-1
    primary_target_x=0
    primary_target_y=0
    primary_target_height=0
    primary_target_speed=0
    traj_x=0
    traj_y=0
    traj_height=0
    traj_speed=0

    -- 诊断状态
    radar_ready=0
    needdup=1
end
```

如果主程序还有 `target0/1/2`、`traj_data`、`traj_uart_bytes`、姿态滤波、门区计数或其他缓存，也必须同步恢复为空状态。

## 7. 主程序接入位置

### 7.1 有效雷达帧接入

主程序完成帧头、长度、帧尾和校验和验证后，先把完整帧交给模块：

```lua
-- frame 是完整二进制雷达帧。
if mode_controller:handle_frame(frame) then
    -- true 表示模块已消费该配置回复，业务解析器无需重复处理。
    return true
end

-- 未被模块消费，继续原有 0x80/0x82/0x86 业务解析。
parse_business_frame(frame)
```

不要把一次 `UartGetRecChAndDel()` 返回的半帧或粘包直接传给模块。必须先由主程序提取一条完整帧。

### 7.2 主循环接入

```lua
while true do
    uart_read_and_parse()

    mode_controller:update(os.time())

    handle_cloud_message()
    maintain_business_state()
    sys.wait(100)
end
```

配置期间仍要继续读取 UART，否则模块无法收到查询回复；只需暂停业务数据更新，不要停止整个主循环。

### 7.3 云端 `sset` 接入

主程序应保存本次请求的 `did`，并在模块完成后回复同一个 `did`：

```lua
local pending_mode_did=nil
local pending_mode_value=nil

local function handle_mode_sset(obj)
    local requested=InstallMode.extract_mode(obj.param)
    if requested==nil then
        send_sset_back(obj.did,1)
        return
    end

    -- 云平台可能在等待期间用同一个 did 重发命令。
    if mode_controller:is_busy() then
        if tostring(obj.did)==tostring(pending_mode_did)
            and requested==pending_mode_value then
            -- 相同请求属于幂等重试，直接忽略，等待原请求完成。
            return
        end
        send_sset_back(obj.did,3)
        return
    end

    pending_mode_did=tostring(obj.did or "")
    pending_mode_value=requested
    local started,rst=mode_controller:request(requested,pending_mode_did)
    if not started then
        send_sset_back(obj.did,rst)
        pending_mode_did=nil
        pending_mode_value=nil
    end
end
```

### 7.4 完成回调与云端回执

```lua
on_install_mode_finished=function(ok,mode,rst,reason)
    local did=pending_mode_did

    if ok then
        -- 只有整个状态机成功后才允许更新并上报 mode。
        current_mode=mode
        mode_verified=1
        send_sset_back(did,0)
        send_mode_dup(mode)
    else
        current_mode=-1
        mode_verified=0
        send_sset_back(did,rst or 2)
        -- reason 已包含失败阶段，例如 boundary_verify_timeout。
        log.warn("MODE", "failed reason="..tostring(reason))
    end

    pending_mode_did=nil
    pending_mode_value=nil
end
```

不要仅根据30°/90°角度查询结果主动上报 `mode`。只有角度、高度、配置清理、边界和自动范围全部成功，才能把模式标记为已验证。

## 8. 云端回复码

| `rst` | 含义 | 建议处理 |
|---:|---|---|
| `0` | 整套配置和回读全部成功 | 更新并主动上报 `mode` |
| `1` | `mode` 缺失或不是0/1 | 检查数据点类型和JSON结构 |
| `2` | UART发送失败或某阶段三次验证失败 | 查看失败阶段和UART日志 |
| `3` | 正在处理另一条不同的切换命令 | 等待原命令完成，不要连续点击 |

同一个 `did + mode` 的重复下发应作为幂等重试忽略，不能返回 `rst=3`；否则云端会同时看到同一 `did` 的忙碌和成功回执。

## 9. 主动上报与查询原则

主程序的 `fill_param()` 或 `sget` 逻辑应只使用最后一次完整成功值：

```lua
if mode_verified==1 then
    dst.mode=mode_controller:get_mode()
end
```

以下情况不得主动上报新模式：

- 只完成角度设置；
- 高度验证失败；
- 配置文件范围清理失败；
- 边界回读不一致；
- 自动范围配置失败；
- 状态机最终返回 `rst=2`。

## 10. 部署检查

1. UART1 设置为115200、8N1。
2. 只有一个任务读取 UART1 和网络通道1。
3. 主任务优先使用 `uart.write(1,data)` 写物理串口。
4. `handle_frame()` 在完整帧校验后调用。
5. `update()` 在主循环中持续调用。
6. 配置期间继续接收 UART。
7. `on_config_start` 已清空所有业务缓存。
8. `ssetbck rst=0` 只在全部阶段完成后发送。
9. 相同 `did + mode` 的重复命令按幂等方式忽略。
10. 不根据单独的角度查询结果更新 `mode`。

## 11. 实物验证

### 11.1 侧装

云端设置：

```json
{"cmd":"sset","did":"side-1","param":{"mode":1}}
```

PC端查询应满足：

```text
安装角度：Z=30.00°
安装高度：220cm
配置文件探测范围：全零、无坐标点
探测范围：手动，X+=165，X-=165，Y+=500，Y-=0cm
```

### 11.2 顶装

云端设置：

```json
{"cmd":"sset","did":"top-1","param":{"mode":0}}
```

PC端查询应满足：

```text
安装角度：Z=90.00°
安装高度：330cm
配置文件探测范围：全零、无坐标点
自动范围模式：开启
```

顶装的 `X±165/Y±250cm` 在开启自动范围前写入并验证。开启自动范围后，`0x07/0x89` 可能返回自动探测点而不再显示固定边界，这是协议表现，不代表前一步未执行。

## 12. 常见故障

| 现象 | 原因 | 处理 |
|---|---|---|
| 收到 `sset` 后雷达参数不变 | UART发送接口未写到物理串口 | 优先使用 `uart.write(1,data)` |
| 只有 `0x82/0x86` 主动帧 | 设置帧未发出或查询未回复 | 检查 `uart tx` 日志和UART通道 |
| 同一 `did` 同时出现 `rst=3`、`rst=0` | 云端重发被当作新请求 | 实现相同 `did+mode` 幂等忽略 |
| `rst=2` 后仍上报新 `mode` | 仅凭角度推断模式 | 只在完整状态机成功后上报 |
| 侧装仍受自动范围影响 | 从顶装切换后限制未关闭 | 明确发送 `0x07/0x08=0` 和 `0x07/0x0C=0` |
| 顶装查询不显示固定外边界 | 已切换为自动范围返回格式 | 查看切换日志中的边界回读阶段 |
| 配置一直处于忙碌 | 主循环没有调用 `update()` 或没有接收UART | 保持100ms循环并持续解析完整帧 |

## 13. 修改后的最低验证要求

```text
□ Lua语法检查通过
□ mode=0、mode=1完整状态机模拟通过
□ 设置帧校验和与协议文档一致
□ 相同 did+mode 重复下发不会返回 rst=3
□ 不同请求在忙碌期间返回 rst=3
□ 任一中间阶段失败不会主动上报新 mode
□ 侧装实物读回30°、220cm和固定边界
□ 顶装实物读回90°、330cm并进入自动范围
```
