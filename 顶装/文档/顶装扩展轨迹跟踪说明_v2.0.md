# 顶装扩展轨迹跟踪说明

当前版本：`Radar_Top_Track_2.0`  
更新时间：2026-07-15

## 1. 文件与部署边界

- 实验入口：`顶装/脚本/lua/radar_iot_task_track.lua`
- 固定版本：`顶装/脚本/lua/versions/track/radar_iot_task_track_v2.0.lua`
- 人数稳定入口仍为 `顶装/脚本/lua/radar_iot_task.lua`，版本 `Radar_Top_Count_1.3`。
- Track 2.0 是独立轨迹扩展，不覆盖 Count 1.3。现场回退人数版时只需重新部署原人数入口。

## 2. 极限口径

“最多轨迹”必须分成四个不同指标：

| 指标 | 当前值 | 含义 |
|---|---:|---|
| 单帧软件解析上限 | 23 | Lua 接受的数据区最大 256 字节，每条目标 11 字节，`floor(256/11)=23` |
| 内部航迹表容量 | 32 | 可同时保存 active、holding 和 candidate 航迹；不是单帧硬件探测能力 |
| 云端完整轨迹槽位 | 3 | 只输出 `track0`、`track1`、`track2`，满足后续三人完整轨迹要求 |
| 雷达硬件实测上限 | 待真机确认 | 既有实物日志观察到约 8 个原始目标；23 是软件模拟上限，不能写成硬件上限 |

内部容量 32 大于单帧上限，是为了在目标短时丢失、重新出现或原始 ID 跳变时保留历史航迹。它不表示雷达能在一帧内识别 32 人。

## 3. 三人完整轨迹

`track0..track2` 使用软件稳定 UID，不直接依赖雷达原始 ID。每个对象包含：

- `uid`、`radar_id`
- `state`、`state_text`：`1/active`、`2/holding`
- `x`、`y`、`height`、`speed`
- `vx`、`vy`、`direction`、`direction_text`
- `hits`、`moving_hits`、`max_disp`
- `path_length`、`path`
- `age_sec`、`lost_sec`、`quality`

软件先按相同雷达 ID 和合理跳变关联；原始 ID 变化时，再使用预测位置、空间距离与高度门限重新匹配。短时消失后航迹进入 holding，重新匹配后恢复 active 并产生 `revived` 事件。路径最多保留最近 12 个采样点，位移达到 8cm 才追加路径点，减少静止抖动。

## 4. 极限诊断字段

| 字段 | 说明 |
|---|---|
| `target_count` | 当前合法轨迹帧中的原始目标数 |
| `raw_target_peak` | 本次 Lua 运行期间出现过的最大原始目标数 |
| `track_table_count` | 当前内部航迹表总数，包含候选和 holding |
| `track_table_peak` | 本次运行期间内部航迹表峰值 |
| `track_table_capacity` | 固定为 32，表示软件容量 |
| `stable_track_count` | 当前范围内 confirmed active/holding 航迹数 |
| `stable_track_peak` | 本次运行期间稳定航迹峰值 |
| `stable_track_data` | 最多前三条稳定航迹的紧凑摘要 |
| `track_event` | 最近生命周期事件：new/confirmed/holding/revived/removed/reset |
| `track_event_seq` | 事件递增序号 |

峰值字段在 Lua 任务重启后重新从 0 统计。`raw_target_peak` 最适合判断雷达一帧实际分离出的最大目标数；`stable_track_peak` 可能包含短时 holding 航迹，适合评估软件维护能力，不应单独当作同时可见人数。

## 5. 真机极限测试方法

1. 部署 `radar_iot_task_track.lua`，确认 `pver=Radar_Top_Track_2.0`。
2. 空场启动 Lua，使所有峰值从 0 开始。
3. 人员逐个进入并分散站位；每一档至少持续 30 秒，同时安排小幅移动，避免所有人完全重叠或被静态过滤。
4. 每档记录人工真值、`target_count`、`raw_target_peak`、`stable_track_count`、`stable_track_peak`、`target_ids`。
5. 每增加 1 人都要确认人数和 UID 至少连续 10 秒；若连续三档增加人工人数但 `raw_target_peak` 不再增加，则记录为当前安装条件下的候选饱和值。
6. 交换人员站位、改变间距后复测。只有多个站位轮次结果一致，才能把该值写为“当前房间和安装参数下的实测上限”。
7. 对最终业务验收，只要求三人时 `stable_track_count>=3`，且 `track0..track2` 的 UID、路径和恢复过程连续。

影响硬件实测上限的因素包括人员间距、遮挡、安装高度、房间多径、软件范围和雷达内部固件算法。因此测试结论必须带现场条件，不能仅写一个无条件人数。

## 6. 上报策略

原始 `target_count`、`target0..target2` 和轨迹签名机制继续保留。轨迹变化立即上报；完全相同的原始轨迹第 1 次上报、连续第 5 次标记稳定，之后每 30 秒保活。该策略只抑制重复 MQTT 数据，不丢弃 UART 原始帧解析，也不停止内部航迹更新。
