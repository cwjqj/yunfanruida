# 顶装单人测试与边缘人员干扰复核

日期：2026-07-14  
现场脚本：`Radar_Top_Count_1.1`  
修正脚本：`Radar_Top_Count_1.2`

版本文件：`顶装/脚本/lua/versions/count/radar_iot_task_v1.2.lua`，与当前部署入口 SHA256 一致。

## 1. 人工真值与环境

- 软件设定范围内实际只有 1 人。
- 探测边缘同一侧另有 3 人。
- 日志范围约 16:46:34 至 16:49:01。

## 2. 关键实测结果

| 字段 | 日志表现 | 与人工真值关系 |
|---|---|---|
| `people_count`（1.1） | 精准帧后为 2 | 多计 1 人 |
| `accurate_people_count` | 2 | 受到边缘人员影响 |
| `target_count` | 3~5 | 包含真人拆分点和边缘人员，不等于人数 |
| `track_people_count` | 1 | 与范围内人工真值一致 |
| `human_present` | 1 | 正常，只表示波束内有人 |
| `door_event_count` | 持续增加 | 边缘人员活动也可能触发，不能用于直接加减人数 |

结论：雷达内部实时/精准人数统计的是其有效波束或内部自动范围，无法严格服从 Lua 软件 ROI。顶装业务人数必须使用软件范围内 confirmed tracks。

## 3. 对日志分析报告的修正

1. `target0/1/2.id` 是雷达原始目标索引，不是 Lua 内部稳定 UID；不能直接声称原始 ID 3 对应内部 UID 5。
2. 同一条 MQTT 记录中，`target_ids` 数量理论上应与 `track_people_count` 一致。若报告出现 `target_ids="5,9"` 但 `track_people_count=1`，需回查原始 JSON 是否来自不同时间点或聚合过程。
3. `-65cm/s` 等于 `-0.65m/s`，属于正常人体移动速度范围，不能仅凭该数值判断为异常跳变。
4. `Y=-137cm` 仍位于当前软件边界 `[-250,250]cm` 内，不属于坐标越界，只表示目标向负 Y 方向移动。
5. `posture_text` 当前本来固定为“未知”；若显示为乱码，属于云端/导出端 UTF-8 解码问题，不代表姿态算法已经输出错误分类。

## 4. 1.2 修正内容

- `COUNT_SOURCE` 默认从 `accurate` 改为 `track`。
- 最终人数只统计当前坐标位于 `X=-165..165cm`、`Y=-250..250cm` 的 confirmed tracks。
- `recompute_people_count()`、`target_ids`、主目标和 `traj_*` 使用同一 `is_countable_track()` 判断。
- 范围外目标仍保留在 `target_count`、`target0/1/2` 中，便于观察边缘干扰。
- 实时人数、精准人数、精准年龄和门事件继续上报，但不改变最终人数。
