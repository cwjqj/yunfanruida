# 顶装姿态识别说明

当前版本：`Radar_Top_Posture_3.1`  
更新时间：2026-07-15

## 1. 文件与版本边界

- 部署入口：`顶装/脚本/lua/radar_iot_task_posture.lua`
- 固定版本：`顶装/脚本/lua/versions/posture/radar_iot_task_posture_v3.1.lua`
- 人数稳定版：`Radar_Top_Count_1.3`
- 扩展轨迹版：`Radar_Top_Track_2.0`
- 姿态识别版：`Radar_Top_Posture_3.1`

三个入口彼此独立。Posture 3.1 从 Track 2.0 派生，但不覆盖 Count 1.3、Track 2.0 或 Posture 3.0 固定副本。姿态只作为稳定航迹的派生属性，不参与人数计算、航迹关联、确认或删除。

## 2. 上床下桌区域模型

当前按现场平面范围划分：

| X 范围 | 区域编码 | 区域文本 | 含义 |
|---|---:|---|---|
| `|X| < 75cm` | 1 | `aisle` | 中间过道 |
| `75cm ≤ |X| ≤ 165cm` | 2 | `bed_desk` | 左右上床下桌区 |
| 房间范围外 | 0 | `outside` | 区域外 |

边界按闭区间执行，因此 `X=±75cm` 和 `X=±165cm` 均属于床桌区。Y 仍使用现有软件房间范围 `-250cm..250cm`；现场坐标若最终改为雷达近端 `Y=0`、向前 `Y=500cm`，需要在独立版本中整体平移 Y 原点，不能只改姿态层。

新增字段：

- `posture_zone`、`posture_zone_text`
- `in_bed_zone`
- `aisle_track_count`、`bed_desk_track_count`
- `bed_profile_ready`、`bed_surface_height`
- `posture_relative_height`

当前 `BED_SURFACE_HEIGHT_CM=0` 表示床面高度未标定。床桌区中高度不低于 145cm 的目标可能是床边站立人员，也可能是上铺躺姿；在床面未标定时，该目标保留候选姿态码，但输出 `posture_valid=0`、`posture_reason="bed_profile_unconfigured"`，并在姿态汇总中计入未知。桌前坐姿等不属于高目标歧义的状态仍可正常输出。

设置床面高度后，脚本预留床面相对高度判断：`15cm ≤ posture_height-bed_surface_height ≤ 70cm` 为上铺躺姿候选，仍需满足严格的躺姿时间和样本确认。该参数必须使用现场实测值，不能采用宿舍家具的通用估计。

## 3. 姿态编码

| 编码 | 英文文本 | 中文含义 |
|---:|---|---|
| 0 | `unknown` | 未知/尚未确认 |
| 1 | `standing` | 站立 |
| 2 | `sitting` | 坐姿 |
| 3 | `lying` | 躺姿 |

协议输出使用 ASCII 文本，避免 DTU 与云平台之间出现中文编码乱码。云端可根据编码显示中文。

## 4. 判定机制

姿态绑定软件稳定 UID，不绑定雷达原始 ID，也不绑定会变化的 `target0..target2` 原始槽位。因此雷达 ID 跳变、短时丢失和恢复后，姿态仍跟随同一软件 UID。

当前初始阈值是待现场标定的地面区参数：

| 场景 | 初始进入阈值 | 稳态迟滞规则 |
|---|---:|---|
| 站 | 高度不低于 145 cm | 已站立时，高度不低于 125 cm 保持站立 |
| 坐 | 70–145 cm | 已坐时，65–150 cm 保持坐姿 |
| 躺 | 高度不高于 70 cm | 已躺时，低于 90 cm 保持躺姿 |

算法保护：

1. 每个 UID 独立保存最近 5 个有效高度，使用中位数过滤单帧尖刺。
2. 新姿态必须同时满足连续样本数和持续时间，不能靠单帧切换。
3. 初始站/坐至少 5 个候选样本且持续 2 秒。
4. 普通姿态变化至少 8 个候选样本且持续 2 秒。
5. 坐到躺至少 10 个候选样本且持续 3 秒；其他进入躺姿至少 12 个候选样本且持续 5 秒。
6. 航迹未 confirmed 时不确认姿态。
7. 航迹进入 holding 后保留最后姿态码，但 `posture_valid=0`；恢复 active 后重新变为有效。
8. 原始 `target0..target2` 始终保持 `posture=0/unknown`，防止把不稳定原始槽误当成某个人。

## 5. 云端字段

`track0..track2` 在 Track 2.0 字段基础上增加：

| 字段 | 含义 |
|---|---|
| `posture` | 姿态编码 0/1/2/3 |
| `posture_text` | `unknown/standing/sitting/lying` |
| `posture_valid` | 当前姿态是否可作为实时有效值 |
| `posture_confidence` | 0–100 的软件置信度 |
| `posture_height` | 最近 5 个有效高度的中位数 |
| `posture_height_valid` | 是否已经采到有效高度 |
| `posture_reason` | `no_track/confirming/confirmed/holding/stale` |
| `posture_candidate` | 正在等待确认的候选姿态编码 |
| `posture_candidate_text` | 候选姿态文本 |
| `posture_age_sec` | 当前稳定姿态已保持秒数 |
| `posture_observed_age_sec` | 距离最近姿态高度样本的秒数 |

独立姿态对象：`posture0`、`posture1`、`posture2`。每个对象带 UID，云端必须按 UID 识别人，不能假设槽位永久固定。

汇总与事件字段：

- `standing_count`、`sitting_count`、`lying_count`、`unknown_posture_count`
- `posture_event`：如 `confirmed:1:unknown:standing`、`changed:2:sitting:lying`
- `posture_event_seq`
- `primary_target_posture`、`primary_target_posture_text`
- `primary_target_posture_valid`、`primary_target_posture_confidence`
- 兼容主目标字段：`posture_code`、`posture_text`

## 6. 能力边界

当前协议只明确每条轨迹包含 ID、size、feature、X、Y、高度和速度，没有说明 size/feature 是姿态码，因此 V3.0 不使用这两个字段推断姿态。

仅凭单一高度，站/坐/躺并不是完全可辨识问题：弯腰或蹲下可能接近坐姿，儿童站立可能接近成人坐姿，地面或家具低反射可能接近躺姿。V3.0 优先输出 `unknown`，不追求过快给出错误结论。

当前没有床区和床面高度标定。若上下铺的床面较高，床上躺姿的绝对高度可能接近地面站姿；未配置床区 profile 前不能可靠区分。Track/Posture 当前有效高度上限仍为 245 cm，上铺坐起目标若超过该值可能先被轨迹层过滤。调整该上限会增加天花板和高处杂波风险，必须结合真机高度分布后单独发布新版本。

## 7. 三脚本统一现场测试

### 7.1 部署确认

分别部署三个文件时检查 `pver`：

| 脚本 | 预期 pver | 主要验收 |
|---|---|---|
| `radar_iot_task.lua` | `Radar_Top_Count_1.3` | 人数 0–3 变化、静止保持、误报清理 |
| `radar_iot_task_track.lua` | `Radar_Top_Track_2.0` | 三个 UID、路径、ID 跳变与恢复 |
| `radar_iot_task_posture.lua` | `Radar_Top_Posture_3.1` | 三人独立站/坐/躺、过道/床桌分区与转换 |

### 7.2 姿态采样

1. 每名测试者分别站、坐、躺，每种状态保持至少 30–60 秒。
2. 记录人工真值、UID、原始 `height`、`posture_height`、姿态码、置信度和有效标记。
3. 至少选择三种不同身高人员重复测试。
4. 三人同时采用不同姿态，并交换位置与姿态，确认姿态不串 UID。
5. 测试两人靠近、交叉、短时遮挡、完全静止和重新进入。
6. 若有上下铺，地面和每层床分别采样，不使用地面阈值直接下结论。

### 7.3 建议验收口径

- 三人稳定后 `stable_track_count>=3`，三个 UID 连续。
- 三个 `track` 对象的 `posture_valid=1`，与人工姿态一致。
- 单帧异常高度不改变稳定姿态。
- holding 时保留姿态但 `posture_valid=0`，revived 后 UID 和姿态连续。
- 姿态变化只在满足确认时间后产生一次 `posture_event`，边界抖动不反复切换。
- 最终阈值必须依据现场高度分布调整；当前阈值不能直接作为硬件规格。
