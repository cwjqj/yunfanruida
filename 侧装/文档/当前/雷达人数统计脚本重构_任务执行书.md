## 0. 背景与目标(Context)

- **硬件链路**:云帆瑞达 R60BMP1 60G 毫米波雷达 → UART1(115200 8N1) → Air780EP 合宙 4G DTU → 银尔达 IOT 平台 MQTT 上云。
- **场景**:学生寝室「上床下桌」,雷达**侧装下倾 30°**、安装高度 220cm、已写入固定探测边界 `X∈[-165,165]cm, Y∈[0,500]cm`。**门/主要出入口实测在雷达坐标 Y≈0、X≈-70(近端偏左)。**
- **为什么要重构**:现有 `radar_iot_task.lua` 用「轨迹目标数约束人数 + 3/5 帧确认 + under_bed 槽位」等一整套**压制式**逻辑,两个致命问题:
  1. **人一旦静止,雷达丢帧 → 目标数掉 → 人数被压低**,与真实不符;
  2. **多帧确认 + 10s 限频拖慢反映速度**,人数变化滞后。
- **实测佐证**:
  - 2026-06-17 日志:精准人数 `0x0C` 长期报 `0 0`、实时人数 `0x0A` 根本不出现 → **人数只能靠轨迹**;并有 `height=23cm` 低矮固定杂波点 → **必须保留高度合理性过滤**。
  - 2026-07-07 正式安装日志:出现**多目标帧**(`data_len` = 33/44/55/66,即 3~6 个目标),再次确认「11 字节/目标」;坐标落在框内,`Y` 最小到 `0x08`(≈8cm,贴近门口 Y≈0)。
- **本次目标(硬指标)**:
  1. **云端 `people_count` 始终等于房间真实人数**;
  2. **轨迹是唯一人数依据**,实时/精准人数仅作辅助字段上报,不参与人数;
  3. 目标走完**完整轨迹后静止**(丢帧)时,**记住其最后坐标继续算一个人**,门口带以外一律**不减人**;
  4. **减人只有两条路径**:目标在**门口带丢失超时**(逐人减)、或**无人信号兜底清零**(整体归零);
  5. **低延迟**:去掉压制式多帧确认,人数变化尽快上云;
  6. **不对轨迹/数据上报做压制限制**。

> ★ **减人机制的关键修正(务必理解)**:雷达**只在固定框 `[-165,165]×[0,500]` 内产生轨迹**。人往门口走、轨迹消失的那一刻,最后坐标**几乎总还卡在框内侧**(如 Y=8),**不会真越界**。所以「严格坐标出界才减人」在本硬件**几乎永不触发** → 多人时走一个不减,只能等全屋空兜底。**因此本版用「门口带 `EXIT_ZONE` + 丢失超时」判离开(见 4.9)。**

## 2. 第一部分:雷达协议与数据解析规范(自包含)

### 2.1 通用帧格式
```
53 59 | Control(1) | Command(1) | Len_H Len_L(2,大端) | Data(n) | Sum(1) | 54 43
```
- **校验 Sum** = 从帧头到数据区(即 `frame[1..6+n]`)逐字节求和取低 8 位。
- **帧尾**固定 `54 43`。
- 解析:找帧头 `53 59` → 读长度 → 等齐 `2+1+1+2+n+1+2` 字节 → 校验和 + 帧尾双校验通过才采用。长度异常(如 `>256`)跳过 1 字节重新同步,防卡死。(现有 `try_parse_frame` 已实现,原样保留)

### 2.2 控制字 / 命令字(本项目用到的)
| 控制字 | 命令字 | 含义 | 数据 |
|---|---|---|---|
| `0x80` | `0x01`/`0x81` | 人体存在 | 1B:1=有人,0=无人 |
| `0x80` | `0x02`/`0x82` | 运动状态 | 1B:0=无,1=静止,2=活跃 |
| `0x80` | `0x03`/`0x83` | 体动参数 | 1B:0-100 |
| `0x82` | `0x02`/`0x82` | **轨迹目标(主数据)** | N×11B,见 2.3 |
| `0x86` | `0x0A` | 实时人数(**仅上报**) | ≥2B:[min,max] |
| `0x86` | `0x0C` | 精准人数(**仅上报**) | ≥2B:[min,max] |
| `0x01` | `0x01` | 心跳 | 记录即可 |

### 2.3 轨迹目标布局:**每目标 11 字节**(关键)
```
id(1) | size(1) | feature(1) | X(2) | Y(2) | height(2) | speed(2)
```
- 数据区长度 = `目标数 × 11`;`data_len==0` 表示当前无目标。
- **执行者须知**:PC 端 `radar_parser.py` 按 **12 字节** 解析是**过时 bug**(会把真实目标误判成「无目标」)。**本脚本坚持 11 字节**,加 `data_len % 11 == 0` 整除保护(不整除告警并跳过)。

### 2.4 坐标 / 速度:**符号位格式**(不是补码!)
X、Y、speed 均「最高位表符号」:最高位 0=正、1=负,低 15 位是幅值。
```lua
local function parse_radar_signed16_be(hi, lo)   -- 原样沿用,勿改成补码
    local val = hi*256 + lo
    if val >= 32768 then return -(val - 32768) end
    return val
end
local function parse_uint16_be(hi, lo) return hi*256 + lo end   -- 高度用无符号
```

## 3. 第二部分:★ x / y / 高度(z) 参数解析准确性验证 ★

> 这是「人数=真实」的地基。解析分**两层**,两层都对人数才可能对。

### 3.1 数值解析层 —— 已用真实帧逐字节验证为**正确**
官方示例帧 + 2026-06-17 真实帧解码(每帧 `data_len=11`,校验 ✓):

| 数据区(Hex) | id | X | Y | 高度z | speed | 说明 |
|---|---|---|---|---|---|---|
| `00 01 02 80 08 00 2E 00 B8 00 06`(官方) | 0 | **-8** | 46 | 184 | 6 | 手册标定值一致 |
| `00 01 02 80 0A 00 34 00 8D 80 00`(真实) | 0 | **-10** | 52 | 141 | **0** | 有人静止 |
| `00 01 02 80 0A 00 34 00 A0 00 1E`(真实) | 0 | -10 | 52 | **160** | 30 | 有人运动 |
| `00 01 02 80 03 00 20 00 95 80 17`(真实) | 0 | **-3** | 32 | 149 | **-23** | 移动中 |

**符号位 vs 补码(证明必须用符号位)**:
| 原始 | 符号位(✅) | 补码(❌) |
|---|---|---|
| `80 0A` | **-10** cm | -32758(荒谬) |
| `80 17` | **-23** cm/s | -32745 |
| `80 00` | **0**(负零) | -32768 |

### 3.2 多目标验证(2026-07-07 正式安装日志)
首帧 `53 59 82 02 00 21 …`(`data_len=0x21=33=3×11`,3 目标)解出:
- 目标1:`id=2, x=44, y=204, h=140, speed=-4`
- 目标2:`id=4, x=-25, y=23, h=168, speed=10`
- 目标3:`id=6, x=-60, y=220, h=153, speed=0`

三目标坐标均落在框内、数值合理 → **多目标 11 字节布局成立**;`y=23` 这类近端点即"人在门口附近"的形态。

### 3.3 「z」的语义澄清(防混淆)
- **高度(z)= 轨迹里的 height 字段**(目标反射点离地高度,cm),目标第三维。
- 安装参数里的 **Z = 安装下倾角 30°**(`0x06/0x01`),是**另一个完全不同的 z**(协议:Z 默认 30° 侧装 / 90° 顶装)。两者不可混用;**姿态判断用 height,不是安装角 Z**。

### 3.4 物理语义层 —— 准确性的**前提条件**(必须现场核对)
雷达输出的 X/Y/height 是按其**内部安装角度(30°)/高度(220cm)**投影出的空间坐标。若**物理实际安装 ≠ 侧装 30° / 220cm**:
- X/Y **系统性偏移** → 「是否在门口带 / 是否在框内」判错 → **减人/保持判错 → 人数错**;
- height 偏移 → 姿态判错。

**标定验证项**(写入脚本头部注释 / 交接文档):
1. 确认物理安装与雷达内部 `0x06/0x01`(角度)、`0x06/0x02`(高度)一致;若顶装须重设角度 90°(`23 28`)。
2. 人站已知位置,核对 `primary_target_x/y` 与卷尺一致(±20cm)。
3. 人站/坐/躺,核对 `primary_target_height` 数量级合理。
4. **★门口标定**:让人从门口进出各走几趟,记录**轨迹消失时的最后 `traj_x/y`**,据此核对/微调 `EXIT_ZONE`(见 5)。

## 4. 第三部分:★ 核心算法 —— 人员航迹保持(Person Track Hold)★

### 4.1 设计原则(对齐 6 条目标)
- `people_count` = 当前**已确认(confirmed)** 且处于 `active`/`holding` 的航迹条数,**唯一权威来源**,重算后钳到 `[0, PEOPLE_COUNT_MAX]`。
- `0x0A`/`0x0C` **只存字段、只上报**,绝不参与 `people_count`。
- **减人只有三条路径**:①航迹在**门口带 `EXIT_ZONE` 内丢失超时**(逐人减);②无人信号**兜底清零**(全清);③单航迹**超长滞留安全上限 `HOLD_MAX_SEC`**(防某个鬼点在有真人时把人数永久钉住)。**门口带以外的静止(床区/两侧墙/远端桌/过道)一律保持不减。**
- **只有走过完整轨迹(`confirmed==true`)的航迹才计数、才享受静止保持**;闪现杂波停在 `candidate`,短时删除、永不计数。
- **删除**全部「按轨迹目标数压制人数」的旧逻辑。

### 4.2 全局状态
```
person_tracks = {}                 -- 航迹数组
uid_seq = 0                        -- 自分配稳定唯一号
all_clear_since = 0                -- 兜底清零计时
accurate_updated_at = 0            -- 精准人数最近到达时刻(兜底用)
last_valid_frame_time = os.time()  -- 看门狗
last_people_count = -1             -- 上报变化检测
```

### 4.3 单条航迹字段
| 字段 | 用途 |
|---|---|
| `uid` | **自分配**稳定唯一号(单调递增,永不复用)。姿态滤波键 / target 槽映射 / 日志。 |
| `rid` | 雷达帧里的目标 id 字节(**易变、会复用**),只作第一优先匹配线索。 |
| `x,y` | 最后(平滑后)坐标。holding 时即"最后坐标",**冻结不清零**。 |
| `height,speed` | 最后高度/速度。 |
| `birth_x,birth_y` | 建轨坐标,算净位移。 |
| `max_disp` | 距出生点的**最大净位移**(cm)。完整轨迹门槛用(比累计路程稳,不被原地抖动累加)。 |
| `hits` | 累计命中帧数。晋级门槛 + 主目标排序。 |
| `first_seen,last_seen` | 建轨 / 最近命中时刻(os.time)。保持/离开/超时计时基准。 |
| `state` | `candidate`(未确认,含"命中过但未达门槛")/ `active`(已确认、在跟)/ `holding`(已确认、丢帧保持)。**三态无 leaving**,离开用时间闸(4.9)。 |
| `confirmed` | 是否过完整轨迹门槛(一旦 true 永久 true);**只有 true 才计入人数,且 `active` 必已 confirmed**。 |
| `in_exit` | 最后坐标是否在门口带 `EXIT_ZONE` 内(每次 bind 重算,决定 holding 是否走离开)。 |
| `close_hits` | 与另一航迹持续贴近的累计帧数(合并去重 `merge_close_tracks` 用)。 |
| `matched_this_frame` | 每帧临时标记,**仅 `on_track_frame` 内用**(防一轨配多点);**maintain 老化不看它**(见 4.9)。 |

派生量(不存字段,实时算):`count = #{ t : t.confirmed and t.state∈{active,holding} }`。

### 4.4 状态机(三态)
```
新点 → candidate ──(完整轨迹门槛 / 静态确认)──→ active ⇄(丢帧≥ACTIVE_TO_HOLD / 再命中) holding
           │(丢帧≥CAND_MISS_TTL → 删,杂波)                              │
           │                                        (门口带内丢帧≥EXIT_GRACE, 或真越界)
           └──────────────────────────────────────────────────────────→ 删除(减人)
    任意状态 ──(兜底清零 / 看门狗清)──→ 全表清空;  holding 丢帧≥HOLD_MAX_SEC ──→ 删除(防鬼点)
```
**只有 `active`/`holding` 且 `confirmed` 计入人数**;`candidate` 不计。

### 4.5 双引擎入口(关键架构)
- `on_track_frame(detections, now)` —— **仅在收到 `0x82/0x02` 帧时**调用:关联 → 更新命中航迹 → 新建 candidate → 合并去重。
- `maintain_tracks(now)` —— **主循环每轮(100ms)都调用,不依赖是否收到帧**:candidate 老化删除、active→holding、holding→离开/超时、兜底清零、重算人数、刷新上报字段。
  > **为什么必须独立**:雷达可能停发帧或发空帧,航迹老化/减人/兜底不能只在收到帧时才跑,否则"人走了但雷达也不发帧了"会永远不减。

### 4.6 on_track_frame(收帧:关联 + 更新 + 新建)
```
on_track_frame(detections, now):        -- detections 已做 11B 解析 + 高度 sanity 过滤
  last_valid_frame_time = now
  for t in person_tracks: t.matched_this_frame = false

  -- Pass A:同 rid 优先(带跳变闸防 id 复用)
  for d in detections where not d.matched and d.rid>=0:
     t = 未被本帧占用的同 rid 航迹
     if t and dist(d,t) <= ID_MATCH_MAX_JUMP_CM: bind(d,t,now)
     -- 同 rid 但位移过大 → 判 id 复用,拒配,留给 Pass B;老 t 按自己规则保持/老化

  -- Pass B:最近距离贪心(全局最小优先,处理 id 抖动/换号)
  收集所有 (d未配, t未配, dd=dist)  where dd<=NEW_MATCH_DIST_CM
  按 dd 升序; 依次 bind(未被占用的配对)

  -- Step C:未匹配检测 → 新建 candidate(防分裂 + 上限)
  for d in detections where not d.matched:
     if #person_tracks >= TRACK_TABLE_MAX: log 丢弃; continue
     if 存在 confirmed 航迹 t 且 dist(d,t)<SPLIT_MERGE_CM: continue   -- 防一人分裂两目标
     uid_seq+=1; 新建 track{uid=uid_seq, rid=d.rid, x=d.x,y=d.y, birth=(d.x,d.y),
                            max_disp=0, hits=1, first_seen=now,last_seen=now,
                            state="candidate", confirmed=false,
                            in_exit=in_exit_zone(d.x,d.y), matched_this_frame=true, ...}

  merge_close_tracks(now)     -- 合并去重(见 4.12)
  -- 注意:老化/离开/兜底/重算/刷新 全部交给 maintain_tracks(主循环每轮),此处不做
```
**空数据区帧**(0 目标):`on_track_frame({}, now)` —— 无匹配、无新建;老化由 maintain 处理,**不再**用旧 `TARGET_CLEAR_CONFIRM_FRAMES` 清表。

### 4.7 bind:更新命中航迹 + 完整轨迹确认
```
bind(d,t,now):
  d.matched=true; t.matched_this_frame=true
  t.x = round(t.x*(1-POS_ALPHA) + d.x*POS_ALPHA)   -- POS_ALPHA=1 即关平滑
  t.y = round(t.y*(1-POS_ALPHA) + d.y*POS_ALPHA)
  t.height,t.speed,t.rid = d.height,d.speed,d.rid
  t.hits += 1; t.last_seen = now
  disp = dist({t.x,t.y},{t.birth_x,t.birth_y}); if disp>t.max_disp: t.max_disp=disp
  t.in_exit = in_exit_zone(t.x,t.y)
  try_confirm(t, now)
  if t.confirmed then t.state = "active" end   -- ★仅"已确认"才 active;未确认一律留 candidate(丢帧走老化);holding 命中→复活 active

try_confirm(t, now):
  if t.confirmed: return
  -- 主口径:走过完整轨迹
  if t.hits>=CONFIRM_HITS and t.max_disp>=CONFIRM_MOVE_CM: t.confirmed=true; return
  -- 静态确认(进门就地坐下/几乎没位移,但确是人):命中够多 + 0x80 有人 + 位置稳
  if STATIC_CONFIRM_ENABLE and t.hits>=STATIC_CONFIRM_HITS
     and human_present==1 and t.max_disp<=STATIC_CONFIRM_STABLE_CM:
     t.confirmed=true
```
> **对应你的需求**:「有完整轨迹路线」= `confirmed`(走够净位移/帧数,或静态确认);「静止后记录最后坐标」= holding 冻结 `x/y/height`;「不更变人数上限」= holding 持续计入不下调。

### 4.8 people_count 重算
```
recompute_people_count():
  n = count(t in person_tracks where t.confirmed and t.state in {active,holding})
  people_count = clamp(n, 0, PEOPLE_COUNT_MAX)
  people_min = people_max = people_count           -- 兼容旧数据点
  has_person = (people_count>0 or human_present==1) and 1 or 0
```

### 4.9 门口带离开判定 + 主循环老化 maintain_tracks(★减人核心)
**门口带 `EXIT_ZONE`(★现场标定项)**:实测门在 **Y≈0、X≈-70**,据此设出口带矩形 `X∈[-120,-20], Y∈[0,70]`(门为中心 ±50cm 宽、近端 70cm 深)。**只圈门这一处**——床/桌沿两侧墙(|X|≈165)、远端(Y≥300)、过道中央一律不算出口,保证躺床上/坐桌前静止丢帧的人**永久保持不减**。
```
in_dorm_area(x,y): AREA_X_MIN<=x<=AREA_X_MAX and AREA_Y_MIN<=y<=AREA_Y_MAX
in_exit_zone(x,y): EXIT_ZONE_X_MIN<=x<=EXIT_ZONE_X_MAX and EXIT_ZONE_Y_MIN<=y<=EXIT_ZONE_Y_MAX

is_leaving(t, now):                                  -- 仅对 holding 调用
  gone = now - t.last_seen
  if gone < EXIT_GRACE_SEC: return false             -- 时间闸,防掉一两帧就误判
  if in_exit_zone(t.x, t.y): return true             -- (主)门口带内丢失超时 → 离开减人
  if STRICT_OUT_OF_ZONE_EXIT and not in_dorm_area(t.x,t.y): return true  -- (兜底)真越界
  return false                                       -- 门口带外静止 → 继续保持
```
**maintain_tracks(now)**(主循环每轮调,倒序遍历便于删除):
```
maintain_tracks(now):
  -- ★老化只看 gone=now-last_seen(last_seen 仅 bind 时更新);不得用 matched_this_frame:
  --   本函数每轮都跑(含"没收到帧"的轮次),那时 matched_this_frame 是上次收帧的残留值,
  --   用它判老化会导致"雷达停发帧后 active 永不转 holding、人数永不减"。
  -- ★remove(i) ≡ remove_track(i):删航迹时**同步 posture_filters[t.uid]=nil**,
  --   否则姿态状态随 uid(单调递增、永不复用)无限累积 → DTU 长期运行内存泄漏。
  for i=#person_tracks..1:
     t=person_tracks[i]; gone=now-t.last_seen
     if t.state=="candidate" and gone>=CAND_MISS_TTL_SEC:
        remove_track(i); continue                    -- 未确认目标丢失(含闪现杂波)→ 删,不计数
     if t.state=="active" and gone>=ACTIVE_TO_HOLD_SEC:
        t.state="holding"                             -- active 必已 confirmed(见 bind);丢帧 → 转保持
     if t.state=="holding":
        if is_leaving(t,now): remove_track(i); continue   -- 门口带丢失超时/真越界 → 减员
        if HOLD_MAX_SEC>0 and gone>=HOLD_MAX_SEC: remove_track(i); continue  -- ★会误删久静止真人!默认 0=关(见参数表警告)
        -- 否则:门口带外静止保持,冻结最后坐标,不减
  update_fallback_zero(now)      -- 见 4.10
  recompute_people_count()       -- 见 4.8
  refresh_target_report()        -- 见 6.3
```
- **holding 保留最后坐标**:holding 不 bind、不清零,`x/y/height/speed/in_exit` 冻结为最后一次 bind 值;target/traj 从航迹表读,云端能看到"这个人最后停在哪"。
- **复活优先**:每帧关联先跑(on_track_frame),holding 被重新命中 → 先回 active、清计时;只有始终没命中的才在 maintain 里老化/删除。防"人还在门口带、掉几帧就被误删又重建"。
- **真正删除只有 4 种**:① candidate 丢失 ≥ `CAND_MISS_TTL_SEC`;② holding 判离开(门口带超时/真越界);③ holding 丢失 ≥ `HOLD_MAX_SEC`;④ 兜底清零全删。删除后 `recompute_people_count` 立即反映。

### 4.10 兜底清零 + 看门狗(在 maintain_tracks 里)
```
update_fallback_zero(now):
  accurate_ok = (accurate_people_max==0) or (now-accurate_updated_at > ACCURATE_FRESH_SECONDS)
  all_clear = (human_present==0 and realtime_people_max==0 and accurate_ok)
  if all_clear:
     if all_clear_since==0: all_clear_since=now
     if now-all_clear_since >= FALLBACK_ZERO_SECONDS:
         person_tracks={}; posture_filters={}; people_count=0; has_person=0; all_clear_since=now; log "fallback clear"
  else: all_clear_since=0

看门狗: if now-last_valid_frame_time >= RADAR_STALE_SECONDS:
         radar_ready=0; log.warn("radar stale")
         if RADAR_STALE_CLEAR: person_tracks={}   -- 默认 false,保守保持,防通信抖动误清
```
> **时序注意**:精准人数 `0x0C` 默认 60s 才来一帧 → `accurate_people_max` 可能滞后;`accurate_ok` 用"过期即视为不阻塞"(`ACCURATE_FRESH_SECONDS`),避免归零被旧值卡死。

### 4.11(逻辑)兜底 vs 逐人 —— 两者互补
- **有人从门口逐个走** → holding 在门口带超时,**逐条减**(4.9);
- **跟踪整体崩溃但屋子确实空了** → `0x80 无人 + 人数=0` 持续 → **整体兜底归零**(4.10)。

### 4.12 合并去重 merge_close_tracks(防「一人分裂两目标」多计)
```
两条航迹持续 MERGE_CONFIRM_FRAMES 帧距离 < MERGE_DIST_CM:
   保留 hits 大者,把 confirmed 归并到保留者,删除弱者
```

### 4.13 边界异常与已知局限(执行者必读)
| 场景 | 处理 / 说明 |
|---|---|
| id 抖动/复用 | 匹配主键用内部 `uid`,不用雷达 `rid`;Pass A 加 `ID_MATCH_MAX_JUMP_CM` 跳变闸,超距判复用拒配,proximity 命中后才更新 `t.rid`。 |
| 目标瞬移/坐标尖刺 | id 匹配被跳变闸挡下→落 proximity;proximity 也超 `NEW_MATCH_DIST_CM`→老航迹不动(继续保持/老化),尖刺点另建 candidate,靠确认门槛过滤。 |
| **紧贴两人合成一个目标** | 只建/命中一条航迹→people_count 可能**少 1**。**这是雷达物理局限,算法无法凭空拆分**;必须诚实披露:紧贴同行的两人可能计为 1,分开后各自成轨恢复。 |
| 一人分裂成两个目标 | `SPLIT_MERGE_CM` 阻止在 confirmed 航迹极近处新建;`merge_close_tracks` 持续贴近则合并。 |
| **进门即坐/静止跟踪不稳** | 静止目标多普勒弱、雷达易丢帧,candidate 可能反复被 `CAND_MISS_TTL_SEC` 删、`hits` 攒不到 `STATIC_CONFIRM_HITS`→**漏计**。缓解:放宽 `CAND_MISS_TTL_SEC`、调低 `STATIC_CONFIRM_HITS`。**这是"人数=真实"最需现场实测调参的一项。** |
| 闪现 1-2 帧杂波 | 停在 candidate,`CAND_MISS_TTL_SEC` 后删,永不计数。 |
| 雷达长时间不出帧 | `maintain_tracks` 独立按 `gone` 老化(不臆造);同时 `0x80 无人+人数=0` 触发兜底;`RADAR_STALE_SECONDS` 告警,`RADAR_STALE_CLEAR` 默认不清(防抖动误清)。 |
| 空轨迹帧(data_len=0) | `on_track_frame({})`:无匹配无新建;老化交给 maintain;不清表。 |
| 人数上限 | `PEOPLE_COUNT_MAX` 钳顶;`TRACK_TABLE_MAX` 封表(杂波风暴拒新建)。 |
| 门口带内静止不走的人 | `EXIT_GRACE_SEC` 时间闸 + 只要被命中就复活;若仍误减,调大 `EXIT_GRACE_SEC` 或缩小 `EXIT_ZONE`。 |
| 单个鬼点在有真人时 | 主要靠**确认门槛**(鬼点难走满完整轨迹)挡在计数外;真人全走后兜底清零收尾。**不建议靠 `HOLD_MAX_SEC`**(默认 0),它会连久静止的真人一起误删。 |
| **整夜睡觉/久坐不动的真人** | `holding` 永久保持(`HOLD_MAX_SEC=0`)。★**切勿把 `HOLD_MAX_SEC` 设成小于在室静止时长**,否则到期误删 → 人数错变 0 且整夜不恢复(0x80 有人→兜底不触发;confirmed 门槛→睡着的人不重建)。 |
| 航迹删除的副作用 | 每处 `remove_track` 必须同步 `posture_filters[t.uid]=nil`;清表处 `posture_filters={}`。否则姿态状态泄漏(见 6.2)。 |
| 11/12 字节固件差异 | `data_len % 11` 自检告警,可快速切 stride。 |

### 6.4 上报节奏(低延迟,不压制)

## 9. 第七部分:验证方法(确保「云端人数 = 真实人数」)

**A. 静态解析自检**(不接雷达,喂固定帧):把 3.1/3.2 的真实帧灌入解析,断言 x/y/height/speed 一致(尤其 `80 0A→-10`、`80 17→-23`、多目标帧拆成 3 个)。

**B. 真机行为用例**(接雷达,看云端 `people_count` 与日志):
| 用例 | 操作 | 期望 |
|---|---|---|
| 单人进入 | 从门口(X≈-70,Y≈0)走入、走动几步 | 走够门槛后 `people_count:0→1`,primary 有坐标 |
| 静止保持 | 坐/躺下不动(雷达丢帧) | `people_count` **保持不掉**;target 显示冻结的最后坐标 |
| 门口离开 | 走到门口带、出房间 | 丢失 ≥`EXIT_GRACE_SEC` 后 `people_count` **减 1** |
| 床区静止不误减 | 一人躺床上(X≈±160)长时间不动 | **保持不减**(门口带不含床区) |
| 多人逐个走 | 3 人在室,1 人从门口离开 | `3→2`,留下的 2 人不受影响 |
