## 1. 项目目标

将云帆瑞达 R60BMP1 60G 毫米波雷达接入 Air780EP 合宙 4G DTU，并通过银尔达/YED IOT 物模型 MQTT 协议上云。

当前使用场景是寝室“上床下桌”，需要实现：

- 是否有人、当前人数、运动状态、体动参数；
- 多目标轨迹：目标编号、X/Y 坐标、高度、速度；
- 主目标和每个目标的姿态：无人/未知、站立、坐下、躺下；
- 结合床+护栏高度约 2.12m，判断床上坐下和床上躺下。

## 5. DTU 主脚本说明

当前主脚本：`radar_iot_task.lua`

### 5.1 通道

| 项目 | 当前值 |
|---|---|
| 串口 | UART1 |
| 波特率 | 115200 |
| 数据格式 | 8N1 |
| 网络通道 | `nid = 1` |
| 主循环周期 | 100ms |

脚本启动后会停掉平台自带协议解析：

```lua
UartStopProRecCh(1)
PronetStopProRecCh(nid)
```

然后由任务脚本手动读取串口和网络通道。

### 5.2 雷达帧

通用帧格式：

```text
53 59 | Control | Command | Len_H Len_L | Data... | Sum | 54 43
```

校验方式：从帧头到数据区求和，取低 8 位。

当前脚本解析：

| 控制字 | 用途 |
|---|---|
| `0x80` | 人体存在、运动状态、体动参数 |
| `0x82` | 轨迹跟踪 |
| `0x86` | 人数统计 |

轨迹目标当前按 11 字节解析：

```text
id(1) + size(1) + feature(1) + x(2) + y(2) + height(2) + speed(2)
```

坐标和速度使用雷达符号位格式，不是普通 int16 补码。当前脚本使用：

```lua
parse_radar_signed16_be(hi, lo)
```

## 7. 姿态判断

当前脚本参数：

```lua
BED_HEIGHT_CM = 212
BED_ON_TOLERANCE_CM = 25
BED_SIT_HEIGHT_CM = 240
TARGET_HEIGHT_MIN_CM = 20
TARGET_HEIGHT_MAX_CM = 245
SIDE_AREA_X_MIN_ABS = 75
SIDE_AREA_X_MAX_ABS = 165
BED_Y_MIN = 0
BED_Y_MAX = 500
BED_SURFACE_Y_MIN = 300
STAND_HEIGHT_CM = 130
SIT_HEIGHT_CM = 70
POSTURE_HYSTERESIS_CM = 10
POSTURE_CONFIRM_FRAMES = 2
PEOPLE_CHANGE_CONFIRM_FRAMES = 3
PEOPLE_ZERO_CONFIRM_FRAMES = 5
```

当前逻辑：

```text
高度 <= 0：
  无人

在左右两侧上床下桌远端床面区域，并且高度 >= 212cm：
  高度越过床上坐/躺阈值并连续确认后：坐下
  否则：躺下

不在床面区域，包括中间过道、床下/桌前近端区域：
  高度越过站立阈值并连续确认后：站立
  高度越过坐姿阈值并连续确认后：坐下
  其他：躺下
```

当前 Lua 已加入稳定处理：

- `0x86/0x0A` 实时人数是平时主导来源：普通变化需连续 3 次确认，归零需连续 5 次确认，确认后更新 `realtime_people_count` 和 `people_count`。
- `0x86/0x0C` 精准人数作为验证/校准来源：如果与实时滤波人数连续冲突 3 次，则校准 `realtime_people_count` 和 `people_count`。
- 轨迹目标先做高度过滤：`20cm <= height <= 245cm` 才算有效目标；只要当前收到轨迹帧，`people_count` 不允许超过有效目标数，用于压制静态杂波导致的实时人数虚高。没有轨迹约束时，精准人数才作为兜底校准来源。
- 主目标从有效目标中选择，优先选择 `in_bed_zone=1` 的床区目标；没有床区目标时回退到第一个有效目标。
- 姿态按目标 ID 保留稳定状态；阈值附近使用 10cm 滞回，并要求连续 2 帧确认后才改变姿态。

### 7.1 2026-07-10 新增轨迹保持版脚本

已按 `雷达人数统计脚本重构_任务执行书.md` 新建 `scripts/lua/radar_iot_task_track.lua`，用于替代当前主脚本做现场验证；未覆盖 `scripts/lua/radar_iot_task.lua` 和 `scripts/lua/radar_iot_task_original.lua`。

轨迹保持版的关键差异：

- `people_count` 只来自 `0x82/0x02` 轨迹解析后的 confirmed 航迹，`0x86/0x0A` 实时人数和 `0x86/0x0C` 精准人数仅作为辅助字段上报，不再校准或压制人数。
- 轨迹帧严格按 11 字节目标布局解析：`id,size,feature,x,y,height,speed`；`x/y/speed` 使用最高位符号格式，不按补码解析。
- 完整轨迹确认后，即使目标静止丢帧也进入 `holding` 并继续计入人数，target/primary/traj 上报冻结的最后宿舍区域坐标。
- 减人只走两类路径：确认航迹在门口带 `EXIT_ZONE_X=[-120,-20], EXIT_ZONE_Y=[0,70]` 内丢失超过 `EXIT_GRACE_SEC`，或全屋无人兜底 `FALLBACK_ZERO_SECONDS` 后清零。
- 动态轨迹上报不再做 10 秒限频；`body_move/traj_x/traj_y/traj_height/traj_speed` 任意变化都会触发 `dup`。
- 保留 `under_bed_people_count=0` 作为云端兼容字段，但不再维护床下槽位，不参与人数计算。

已做的离线验证：执行书样例 `80 08` 可解析为 `-8cm`；多目标样例可解析为 `id=2,x=44,y=204,h=140,s=-4`、`id=4,x=-25,y=23,h=168,s=10`、`id=6,x=-60,y=220,h=153,s=0`。当前电脑未安装 `lua/luac/luajit`，尚未做 Lua 解释器语法检查；需要在 DTU 任务环境或安装 Lua 后再做运行验证。

当前场景按房间宽 3.3m、中间过道 1.5m 处理：`|X|<75cm` 为过道，`75cm<=|X|<=165cm` 为左右两侧上床下桌区域。床面区域进一步要求 `Y>=300cm`；`Y<300cm` 先按床下/桌前近端处理。床区判断在 Lua 软件层完成，并上报 `primary_in_bed_zone` / `targetN.in_bed_zone` 给云端；雷达硬件参数建议保持能稳定出轨迹的原始范围，不再写床标签或固定边界。

## 9. 上报节奏

| 条件 | 行为 |
|---|---|
| 开机 | 延时 2 秒后发送 `dreg` |
| 雷达未就绪 | 不立即 `dup`，除非开机超过 30 秒 |
| 关键状态变化 | 触发 `dup` |
| 动态数据变化 | 限频 10 秒触发一次 `dup` |
| 无交互超时 | 180 秒触发一次上报 |
| 主循环 | 每 100ms 轮询 |

动态变化阈值：

| 字段 | 阈值 |
|---|---:|
| `body_move` | 10 |
| `traj_x` | 20cm |
| `traj_y` | 20cm |
| `traj_height` | 20cm |
| `traj_speed` | 10cm/s |

轨迹保持版 `radar_iot_task_track.lua` 为了满足“不限制轨迹上报”，动态字段变化不再使用上述阈值和 10 秒限频；任意轨迹坐标、高度、速度或体动变化都会触发 `dup`。180 秒心跳、开机 2 秒 `dreg`、雷达未就绪时开机 30 秒强发的框架仍保留。
