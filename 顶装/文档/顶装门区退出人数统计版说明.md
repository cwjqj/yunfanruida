# 顶装门区退出人数统计版说明

日期：2026-07-14  
独立脚本：`顶装/脚本/lua/radar_iot_task_door_exit.lua`  
离线测试：`顶装/脚本/lua/tests/door_exit/test_radar_iot_task_door_exit.lua`  
版本：`Radar_Top_Door_Exit_1.0`

## 1. 目的与边界

本版本用于解决 confirmed tracks 在人员离开后长期保持、现场实际最多 5 人但内部航迹可能累积到 6~8 人的问题。

原脚本 `顶装/脚本/lua/radar_iot_task.lua` 不修改、不覆盖。本版本单独部署和验证，出现误减时可立即切回原脚本。

本版本以 confirmed tracks 作为 `people_count` 来源。雷达实时人数和精准人数只作为差异校验，不能直接创建、增加或删除人员。

同时保留当前主脚本 `Radar_Top_Count_1.2` 的顶装 ROI 规则：只有坐标仍位于 `X=-165..165cm`、`Y=-250..250cm` 内的 confirmed track 才计入人数；范围外原始目标仍通过 `target0/target1/target2` 保留用于干扰排查。

## 2. 软件门标签

2026-07-12 顶装实物记录中，模块在设置 90° 安装角后自动生成的门标签参考为：

| 参数 | 当前值 |
|---|---:|
| 中心 | `X=-175cm, Y=0cm` |
| 宽度 | `150cm` |
| 高度/深度 | `200cm` |

独立 Lua 当前使用对应矩形边界：

```text
X = -250 .. -100cm
Y = -100 ..  100cm
```

这些值是已有实物记录的起始值，不是最终现场验收值。正式部署前必须让人员从真实门口进出多次，记录轨迹消失前最后的 `target0/target1/target2.x/y`，再缩小或平移门区。

## 3. 删除一人的条件

一条航迹必须满足以下顺序，才允许从最终人数中删除：

1. 航迹已通过命中次数、移动距离和运动命中条件，成为 confirmed track。
2. 航迹曾经出现在房间核心区，证明它不是刚在门口生成的临时点。
3. 航迹随后连续至少 2 次进入软件门标签区域，设置 `door_armed=true`。
4. 航迹在门区附近丢失并进入 holding。
5. 满足以下任一确认条件：
   - 最近 6 秒内有真实运动，且丢失达到 3 秒；
   - 模块上报 `0x86/0x14` 出门事件，且该航迹在门区丢失；
   - 模块原始人数或配置上限持续低于内部航迹数 5 秒，且该航迹是门侧 holding 航迹。

删除操作针对具体内部 UID，执行 `table.remove(tracks,index)`，随后重新计算 `people_count` 和 `target_ids`。不是只在显示层把人数强制减一。

## 4. 防误减规则

- 房间核心区内丢轨只转为 holding，不按普通超时删除。
- 门口新生成、从未进入房间核心区的候选点不能触发退出删除。
- 靠近门后又返回房间核心区时，立即取消 `door_armed`。
- 门事件最多匹配删除一条符合条件的门侧航迹，不直接对人数执行 `people_count-1`。
- 实时/精准人数差异只能清理门侧 holding 航迹，不能删除房间内部 active 航迹。
- 现场物理上限当前设置为 `MAX_EXPECTED_PEOPLE=5`，但超过 5 时也只允许删除已到门侧并进入 holding 的幽灵航迹，不对输出做简单截断。

## 5. 硬件门标签

软件门区退出判断不依赖模块门事件，因此即使模块没有标签，也能根据轨迹最终坐标工作。

若需要同时使用模块 `0x86/0x14` 进出门事件，配置 90° 顶装参数时不要清除模块自动生成的门标签：

```powershell
python .\顶装\脚本\python\configure_dorm_bunk_radar.py --port COM3 --keep-labels
```

执行后应查询标签并确认门标签仍存在。若现场门位置与自动标签不一致，应先完成坐标采样，不要直接依赖门事件方向。

## 6. 云端数据点

本版本复用现有数据点，不要求新增物模型：

- `people_count` / `people_min` / `people_max`：confirmed tracks 最终人数。
- `track_people_count`：内部确认航迹数量，与最终人数一致。
- `target_ids`：仍被计入人数的内部 UID。
- `primary_target_*` / `traj_*`：当前主内部航迹。
- `target0` / `target1` / `target2`：最新原始轨迹帧的前三个目标。
- `realtime_people_*` / `accurate_people_*`：模块参考人数。
- `door_event` / `door_event_count` / `door_event_at`：模块门事件诊断。
- `uart_bytes`：最近一条通过校验的完整帧。

## 7. 离线验证

运行：

```powershell
cd .\顶装\脚本\lua
lua .\tests/door_exit/test_radar_iot_task_door_exit.lua
```

当前覆盖：

1. 室内确认航迹丢失后保持 1 人。
2. 航迹从房间核心区返回门区并丢失后变为 0 人。
3. 5 人中 1 人从门区离开，最终保留 4 人。
4. 门标签出门事件只删除一条符合条件的航迹。
5. 6 条内部航迹与模块稳定精准人数 5 不一致时，只清理门侧幽灵航迹，最终保留 5 人；没有可靠模块人数时，配置上限 5 作为同类兜底参考。

通过输出：

```text
PASS: indoor loss retained, door exit removed, five-person exit became four, door event matched one track, door ghost pruned to max five
```

## 8. 真机验证顺序

1. 空场运行 5 分钟，确认无虚假 confirmed track。
2. 单人从门进入并停留，确认不会在门区刚生成时误删。
3. 单人在房间内部静止并制造短时丢轨，确认人数保持。
4. 单人从门离开 5 次，记录每次最后坐标和减人延迟。
5. 5 人全部进入，再逐个离开，确认 `5→4→3→2→1→0`。
6. 对比 `track_people_count`、精准人数和人工真值，检查是否仍有门侧幽灵 UID。

现场验收前最大的风险仍是门区坐标未经最终安装位置复核。门区过大会误删靠近左侧墙体的人，门区过小则不能删除真正离开的航迹。
