# 雷达人数统计交叉验证与最终规则

## 1. 现场事实与问题

- 现场事实：昨天夜间床下/下面区域实际只有 1 人；这是昨天这批导出数据的人工真值，用于判断导出数据是否虚高，不是长期人数上限。
- 云端导出问题：`2026-07-10 02:00-08:00` 大量记录显示 `当前人数=3/4`，且 `当前人数` 与 `目标数量` 高度同步。
- 行为问题：昨天代码不能稳定识别人出去并减人；数据上报延迟明显；雷达断电或断串口后云端仍保留旧人数。

## 2. 三版代码交叉验证

### 2.1 `radar_iot_task_original.lua`

主要问题：

- `0x86/0x0A` 或 `0x86/0x0C` 到达时直接把 `people_count` 改成雷达人数统计值。
- `0x82` 轨迹目标到达时，如果 `target_count > people_count`，会把 `people_count` 抬高到目标数量。
- `0x80` 无人时直接清人数和目标，但对“人出去后雷达还残留目标/云端旧值”的情况处理不足。

结论：原始版会把雷达目标数和雷达内部人数统计直接当真实人数，无法用于床下最多 1 人的场景。

### 2.2 `radar_iot_task.lua`

主要问题：

- 虽然增加了实时/精准人数多帧确认，但最终仍通过 `set_standard_people_count()` 混合 `realtime_people_count`、`accurate_people_count`、`target_people_count`。
- `accept_target_people_count(target_count)` 会把目标数量稳定后写入人数约束。
- 轨迹帧空帧会走 `target_clear_hits` 和清目标逻辑，但不能解决“固定反射/同一人拆成多个目标”的虚高。
- `under_bed_slots` 是附加床下槽位，不是主人数依据；它不能纠正主 `people_count` 被目标数量抬高的问题。

结论：当前版比原始版稳，但仍把 `target_count` 引入人数，和昨天 Excel 中 `当前人数≈目标数量` 的错误模式一致。

### 2.3 `radar_iot_task_track.lua` 初版

优点：

- `0x86` 实时/精准人数只上报，不再参与 `people_count`。
- `people_count` 改为 confirmed 航迹数。
- 人静止丢帧后进入 `holding`，不会因为目标暂时消失就掉人数。

仍需修正：

- `STATIC_CONFIRM_ENABLE=true` 时，稳定反射点只要命中足够多且 `human_present=1`，仍可能被确认为人。
- 下面区域一个人被拆成多个目标时，多个航迹仍可能叠加成人数。
- `RADAR_STALE_CLEAR=false` 时，雷达断电/断串口后云端可能长期保留旧人数。
- 开机后只有 `radar_ready=1` 或 30 秒后才 `dup`，会造成云端旧值延迟刷新。

## 3. 最终人数确认规则

最终脚本以 `scripts/lua/radar_iot_task_track.lua` 为准。

### 3.1 什么可以确认“有人”

只有满足完整轨迹条件的 `0x82` 航迹才可以进入人数：

- 目标按 11 字节轨迹格式解析成功。
- 高度在 sanity 范围内：`20cm <= height <= 245cm`。
- 经过内部航迹关联后，命中帧数达到 `CONFIRM_HITS`。
- 从出生点到当前点的最大净位移达到 `CONFIRM_MOVE_CM`。

### 3.2 什么不能确认“有人”

以下数据只能上报或辅助判断，不能增加 `people_count`：

- `0x86/0x0A` 实时人数。
- `0x86/0x0C` 精准人数。
- 原始 `target_count`。
- 单帧闪现目标。
- 长时间稳定但没有完整位移轨迹的目标。

### 3.3 床下/下面区域规则

根据昨天导出数据，下面区域存在明显的一人多目标拆分；算法只能合并疑似同一人的近距离目标，不能硬限制下面人数：

- `LOWER_AREA_Y_MAX=300`，默认把 `Y<300cm` 视作下面/近端计数区域。
- `LOWER_AREA_SPLIT_MERGE_CM=110`，下面区域目标在该距离内优先合并，防止一个人被拆成多个目标。
- 该规则只作用于 `people_count` 的航迹合并和确认，不限制 `target0/1/2` 原始目标字段上报。
- 真实有多人进入下面区域时，应由多条彼此距离足够远、各自满足完整轨迹条件的航迹计入人数。

### 3.4 人出去如何减人

减人路径有三类：

- 确认航迹最后坐标在门口带 `EXIT_ZONE_X=[-120,-20]`、`EXIT_ZONE_Y=[0,70]`，丢失超过 `EXIT_GRACE_SEC` 后减人。
- 下面区域航迹刚发生过运动，随后丢失超过 `LOWER_MOVING_LOST_EXIT_SEC` 后减人，用于补足“没走到门口带但实际离开/消失”的情况。
- 全屋无人兜底：`human_present=0`、实时人数为 0、精准人数不阻塞，持续 `FALLBACK_ZERO_SECONDS` 后清零。

### 3.5 断电/断串口如何清旧值

- `RADAR_STALE_CLEAR=true`：超过 `RADAR_STALE_SECONDS` 未收到有效雷达帧时清航迹和人数。
- `force_initial_dup=1`：脚本启动后第一次 `dup` 不等待 `radar_ready`，先把当前本地人数状态推到云端，避免云端继续显示断电前旧人数。
- 如果是 DTU 整机断电，脚本无法继续发包；云端旧值只能在 DTU 重新上电后的首包 `dup` 被覆盖。

## 4. 最终代码落点

| 规则 | 文件位置 |
|---|---|
| 关闭静态确认 | `scripts/lua/radar_iot_task_track.lua` 的 `STATIC_CONFIRM_ENABLE=false` |
| 下面区域防拆分 | `LOWER_AREA_Y_MAX`、`LOWER_AREA_SPLIT_MERGE_CM` |
| 人数只由航迹重算 | `recompute_people_count()` |
| 下面区域防分裂 | `split_merge_distance()`、`enforce_lower_area_cap()` |
| 人出去减人 | `maintain_tracks()` 中 `exit_zone_timeout` 与 `lower_moving_lost` |
| 断帧清旧人数 | `RADAR_STALE_CLEAR=true` |
| 开机覆盖云端旧值 | `force_initial_dup=1` |
| 原始串口帧上报 | `uart_bytes` 上报最新一次 UART 接收数据的 HEX 文本 |

## 5. 验证重点

下一次现场验证只看 `people_count` 是否匹配人工真值，不对人数做固定上限：

1. 若现场床下实际 1 人，`target_count` 可大于 1，但 `people_count` 应为 1。
2. 若现场实际多人进入下面区域，只有彼此分离且各自满足完整轨迹条件时，`people_count` 才应增加。
3. 人从床下离开：若出现运动后丢失，`LOWER_MOVING_LOST_EXIT_SEC` 后应减人。
4. 人从门口离开：门口带丢失超过 `EXIT_GRACE_SEC` 后应减人。
5. 雷达断电或拔串口：超过 `RADAR_STALE_SECONDS` 后应主动清 `people_count` 并上报。
6. DTU 重新上电：首包 `dup` 应覆盖云端旧人数。
