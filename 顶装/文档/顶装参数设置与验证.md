# 顶装雷达参数设置与验证

更新时间：2026-07-14  
适用模块：云帆瑞达 R60BMP1 60G 毫米波雷达  
安装环境：与侧装一致，学生寝室“上床下桌”，长 500cm、宽 330cm；雷达安装在宿舍中心，高度 330cm

## 1. 配置入口

脚本：`顶装/脚本/python/configure_dorm_bunk_radar.py`

```powershell
# 离线预览，不打开串口、不写设备
python .\顶装\脚本\python\configure_dorm_bunk_radar.py --dry-run

# 正式下发，每一步设置后立即查询并校验读回值；默认使用自动范围
python .\顶装\脚本\python\configure_dorm_bunk_radar.py --port COM3

# 仅在明确需要固定边界时切换为固定模式
python .\顶装\脚本\python\configure_dorm_bunk_radar.py --port COM3 --set-fixed-range
```

依赖 `pyserial`，串口默认 `COM3 / 115200 / 8N1`。

### 1.1 DTU Lua 解析入口

脚本：`顶装/脚本/lua/radar_iot_task.lua`

当前版本：`Radar_Top_Count_1.3`

版本文件：`顶装/脚本/lua/versions/count/radar_iot_task_v1.3.lua`

版本管理约定：无版本号文件始终作为当前部署入口；每次功能更新同时保存 `radar_iot_task_v<版本号>.lua`，并在脚本 `pver`、测试记录和交付说明中写明同一版本号。历史版本文件不再覆盖修改。

该脚本沿用侧装现场版已经验证的 UART 收帧、校验和、11 字节轨迹解码、内部航迹关联及 MQTT 数据点字段。顶装默认房间边界为 `X=-165..165cm`、`Y=-250..250cm`。

原始轨迹与内部稳定航迹分开上报：`target_count` 表示当前原始 `0x82/0x02` 帧中的目标数量；`target0`、`target1`、`target2` 分别保存前三个原始目标。`target_ids`、`primary_target_*` 和 `traj_*` 来自范围内 confirmed tracks。`Radar_Top_Count_1.3` 继续以范围内 confirmed tracks 作为最终 `people_count`；雷达实时/精准人数保留为全波束诊断值。

`target0`、`target1`、`target2` 均为 JSON 对象，格式与侧装历史数据点一致：

```json
{
  "id": 0,
  "x": 116,
  "y": 77,
  "height": 150,
  "speed": 3,
  "posture": 0,
  "posture_text": "未知",
  "in_bed_zone": 0
}
```

当前初始版本只实现原始坐标解析，尚未扩展姿态和床区识别，因此有目标时固定使用 `posture=0`、`posture_text="未知"`、`in_bed_zone=0`。无目标槽位使用 `id=-1`、数值字段清零、`posture_text="无人"`，避免云端残留上一目标。

三目标轨迹帧的数据长度应为 `3 × 11 = 33` 字节，即长度字段为 `00 21`。完整帧通过校验后，数据区按以下方式切分：

```text
目标 1：数据区第  1~11 字节 -> target0
目标 2：数据区第 12~22 字节 -> target1
目标 3：数据区第 23~33 字节 -> target2
```

串口接收不以一次 `UartGetRecChAndDel()` 的返回内容作为帧边界。半帧会留在 `rx_buf` 中等待后续字节；粘包则在解析完第一条完整帧后继续解析缓存中的下一帧。只有帧头、长度、校验和和帧尾全部有效时，才更新上述目标参数。

门区和特殊快速丢失区尚未完成现场标定，因此 Lua 中两个区域开关默认关闭，避免套用侧装方向和门区后产生误退出。现场采样完成后，只在脚本开头“顶装现场标定参数”区域填写边界并启用；目标高度和航迹阈值目前也暂沿用侧装验证值，需要通过空场、单人和多人测试继续校准。

Lua 继续复用侧装云端数据点名称。其中 `uart_bytes` 上传最近一条通过长度、校验和及帧尾检查的完整雷达串口帧，内容为以空格分隔的 HEX 字符串。每解析到新完整帧，`radar_frame_count` 都会递增，但不再由帧计数单独触发上报。原始轨迹变化立即上报；完全相同的轨迹在第 1 帧和第 5 帧上报，第 5 帧标记稳定，之后每 30 秒保活。串口半帧仍先保留在本地接收缓存，拼成完整帧后才更新数据点。

### 1.2 顶装人数统计口径（`Radar_Top_Count_1.3`）

`Radar_Top_Count_1.1` 曾默认使用 `COUNT_SOURCE="accurate"`。单人现场测试时，设定范围内人工真值为 1，但探测边缘同侧另有 3 人，雷达精准人数上报为 2；与此同时 `track_people_count=1`。这说明雷达内部人数属于整个有效波束的统计，不能可靠代表软件设定区域内的人数。

因此 1.2 默认人数源改为 `COUNT_SOURCE="track"`：

- `0x82/0x02` 范围内 confirmed tracks：产生最终 `people_count`、`track_people_count`、`target_ids` 和主目标轨迹。
- `0x86/0x0A` 实时人数：保留原始值，用于观察雷达整个波束内的快速人数变化。
- `0x86/0x0C` 精准人数：保留原始值和年龄，用于和范围内人数交叉诊断，不直接覆盖最终人数。
- `0x86/0x14` 进出门事件：只记录 `0=进、1=出`，不直接对人数执行 `+1/-1`。
- `target_count`、`target0/1/2`：继续显示范围内外的原始检测点，便于识别边缘干扰。

范围过滤规则：

1. confirmed track 的当前平滑坐标必须位于 `X=-165..165cm`、`Y=-250..250cm` 才计入人数。
2. `recompute_people_count()`、`target_ids` 和主目标选择共用同一个范围判断，避免人数为 1 但稳定 UID 列表显示 2 人。
3. 范围外 confirmed tracks 不计数、不进入 `target_ids`，但对应原始目标仍保留在 `target0/1/2` 和 `target_count` 中。
4. `people_count_valid` 在雷达正常收到有效帧时为 1；`people_count_pending` 在 track 模式固定为 0。
5. 整个雷达超过 30 秒没有任何有效帧时，清空人数、实时/精准人数和目标，并上报 `radar_ready=0`。

1.3 将航迹丢失清理改为分级处理：candidate 仍在 2 秒后清理；未达到 mature 条件（少于 12 次命中或存活不足 30 秒）的 confirmed track，在全空信号持续 20 秒后清理；达到 mature 条件的 confirmed track 最长保持 300 秒。已标定的门区或特殊快速丢失区可在移动消失 8 秒后清理，但顶装当前两个区域开关默认关闭。15 秒全量 `fallback_zero` 已移除，不再一次性清空成熟多人航迹。

新增顶装数据点：

| 数据点 | 类型 | 说明 |
|---|---|---|
| `people_count_valid` | 整数/枚举 | track 模式下跟随 `radar_ready`，表示当前范围人数链路是否有效 |
| `people_count_pending` | 整数/枚举 | track 模式为 `0`；切换 accurate 回退模式时才表示等待精准确认 |
| `people_count_source` | 字符串 | 顶装 1.3 默认 `track`；`accurate` 仅保留为回退选项 |
| `track_people_count` | 整数 | 范围内 confirmed tracks 数量，与最终 `people_count` 一致 |
| `accurate_age_sec` | 整数 | 距离最后精准帧的秒数；未收到或雷达超时后为 `-1` |
| `door_event` | 整数/枚举 | `-1`=尚无/已清空，`0`=进，`1`=出 |
| `door_event_count` | 整数 | 当前任务运行期间收到的合法门事件累计数 |
| `door_event_at` | 整数 | 最近门事件的 `os.time()` 秒时间戳 |
| `traj_repeat_count` | 整数 | 当前原始轨迹签名连续出现次数；签名变化后从 1 重新计数 |
| `traj_stable` | 整数/枚举 | 相同原始轨迹连续达到 5 帧后为 1，轨迹变化后恢复为 0 |
| `traj_changed` | 整数/枚举 | 本次原始轨迹签名是否变化；变化帧为 1 |
| `traj_signature` | 字符串 | `target_count` 与前三个原始目标槽位组成的去重签名 |

云平台使用这些字段前，需要创建对应数据点并保证类型一致；否则至少保留 DTU 日志用于判断“当前人数已确认”还是“正在等待精准帧”。

### 1.3 独立扩展轨迹版（`Radar_Top_Track_2.0`）

扩展轨迹实验入口为 `顶装/脚本/lua/radar_iot_task_track.lua`，固定版本副本为 `versions/track/radar_iot_task_track_v2.0.lua`。它不覆盖人数稳定入口 `radar_iot_task.lua`（Count 1.3）。

Track 2.0 的软件单帧完整解析上限为 23 条目标：数据区限制 256 字节，每目标 11 字节。内部航迹表容量为 32，用于同时保留 active、holding 和 candidate 航迹；云端仅输出 `track0..track2` 三条完整稳定轨迹。离线构造 23 目标连续移动帧已通过，但这只是软件解析上限；雷达硬件在本房间的真实分离上限仍须逐人真机测试。

新增极限诊断字段为 `raw_target_peak`、`stable_track_peak`、`track_table_count`、`track_table_peak`、`track_table_capacity`。详细字段、三人连续性和真机测试步骤见 `顶装/文档/顶装扩展轨迹跟踪说明_v2.0.md`。

### 1.4 独立姿态识别版（`Radar_Top_Posture_3.1`）

姿态实验入口为 `顶装/脚本/lua/radar_iot_task_posture.lua`，当前固定版本副本为 `versions/posture/radar_iot_task_posture_v3.1.lua`，并保留 `v3.0` 回退版。它从 Track 2.0 派生，但不覆盖人数和轨迹两个稳定入口。

Posture 3.1 对三个软件稳定 UID 分别维护站、坐、躺状态，采用 5 点高度中位数、阈值迟滞、连续样本和持续时间共同确认。`|X|<75cm` 为过道，`75cm≤|X|≤165cm` 为左右上床下桌区。床面高度未标定时，床桌区高目标输出 `posture_valid=0`，避免把上铺躺姿误当成可靠站姿。详细字段与统一测试方法见 `顶装/文档/顶装姿态识别说明_v3.1.md`。

## 2. 默认参数

| 参数 | 默认值 | 说明 |
|---|---:|---|
| 安装角度 X/Y/Z | `0° / 0° / 90°` | 顶装向下；角度按 100 倍大端传输 |
| 安装高度 | `330cm` | 宿舍中心顶装实际高度 |
| 探测范围模式 | 自动 | 模块实际使用自动范围 |
| 自动范围外边界 | `X±165cm, Y±250cm` | 先写入宿舍实际边界，再开启自动范围限制 |
| 人体存在 | 开启 | 设置后查询验证 |
| 轨迹跟踪 | 开启 | 设置后查询验证 |
| 实时人数间隔 | `1s` | 与侧装一致 |
| 精准人数间隔 | `60s` | 与侧装一致 |
| 标签 | 全部清除 | 90°角度会自动生成门标签，随后统一清除 |
| 轨迹产生距离阈值 | `50cm` | 缩短轨迹生成距离 |
| 误报点消除时长 | `30s` | 与侧装一致 |
| 存在轨迹时间 | `2s` | 与侧装一致 |

顶装脚本不下发干扰边 `0x07/0x1A`。厂商设置指南注明该功能只适用于门上方中央倾斜 30° 安装模式。
